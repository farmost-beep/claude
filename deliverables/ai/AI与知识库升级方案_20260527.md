## 一、突破点1：量化策略→投资决策管道

### 1.1 为什么选这个

你已经有了所有需要的零件——autostrategy（策略生成引擎）、19个金融插件（数据分析能力）、13位大师框架（投资哲学指导）。但这些零件还没有串联成一条能产出可操作投资洞察的生产线。

**你不需要成为Quant**。你需要的是：用你的金融判断力挑选策略方向 → AI生成策略代码并回测 → 你用银行信贷的视角审视策略的风险侧 → 筛选出真正适合自己资金体量和风险偏好的策略。

### 1.2 目标

到2026年12月，建成一个**个人量化策略库**，内含3-5个经过回测验证的策略，每个策略有明确的适用范围、风险边界和你的个人适配评估。

### 1.3 路径（6个月，每月一个里程碑）

#### 第1个月（7月）：跑通第一条策略管线

**任务**：用autostrategy生成1个基于大师框架的策略，跑通"设计→代码→回测→审查"全流程。

**具体操作**：
1. 从13位大师中选1位你最认可其理念的（建议从格雷厄姆或巴菲特入手，因为你有成熟的银行信贷视角来审视他们的"安全边际"和"护城河"概念）
2. 用autostrategy将大师的投资逻辑转化为量化策略
3. 回测该策略在A股近3年的表现
4. 用你的银行信贷经验审阅：这个策略在哪些市场环境下会失效？为什么？

**产出**：
- 1个完整的策略包（STRATEGY_DESIGN.md + strategy.py + 回测报告）
- 1份"我的审阅笔记"：标注策略的隐含假设、失效条件、与大师原文的偏差

**验证标准**：你能用自己的话解释这个策略的买卖逻辑、为什么回测中某些时段表现好/差。

#### 第2个月（8月）：19个插件融入投研管线

**任务**：建立"插件→策略设计"的标准工作流，让金融插件为策略生成提供数据弹药。

**具体操作**：
1. 设计一个标准化流程：
   ```
   选行业 → /sector 行业名（market-researcher 行业概览）
        → /screen（筛选标的）
        → /financial-analysis（财务建模）
        → autostrategy（量化策略生成）
        → /valuation-reviewer（估值审查）
   ```
2. 选一个你工作关系已熟悉的行业（光伏或半导体），跑通这个流程
3. 对比"有插件辅助的策略"vs"纯人工判断的策略"在数据覆盖度上的差异

**产出**：
- 1份"插件→策略标准工作流"文档（含插件调用清单和决策节点）
- 1个行业策略（光伏或半导体）

**验证标准**：你开一个新对话，给一句话指令，Claude Code自动执行"插件分析→策略生成"全流程。

#### 第3个月（9月）：策略风险压力测试——银行级审阅

**任务**：这是你的绝对主场。用银行信贷审批的视角，对生成的策略做"贷后管理式"的风险审查。

**具体操作**：
1. 对前两个月生成的2个策略，做一份"策略信贷评估报告"：
   - **现金流视角**：策略的保证金/持仓周转是否会产生隐性的资金压力？
   - **集中度风险**：策略是否过度集中于某个行业或市值区间？
   - **极端情景**：假设出现2015年股灾/2020年疫情级别的冲击，策略最大回撤会是多少？
   - **对手方风险**：策略依赖的数据源/交易通道出问题时，怎么处置？
2. 给每个策略一个"授信评级"：AAA（可实盘追随）/ AA（可观察跟踪）/ A（需优化）/ B（暂不适合）

**产出**：
- 1份"策略信贷评估报告"（这是银行人对量化策略的独特视角，也是你最独特的价值贡献）
- 每个策略的风险分级

**验证标准**：报告中至少包含一个你基于银行业务经验发现的、纯量化回测不会暴露的风险点。

#### 第4个月（10月）：建立策略库与追踪系统

**任务**：不再零散地生成策略，而是建立结构化的策略库和追踪机制。

**具体操作**：
1. 建立策略库目录结构：
   ```
   strategies/
   ├── active/          # 当前追踪中的策略
   │   ├── graham_value/  # 格雷厄姆价值策略
   │   └── solar_rotation/ # 光伏轮动策略
   ├── archived/        # 已验证但暂不适用的策略
   ├── templates/       # 策略模板
   └── strategy_index.md # 策略索引表
   ```
2. 每个策略目录固定包含6个文件：DESIGN.md / strategy.py / backtest.json / risk_assessment.md / tracking_log.md / decision_journal.md
3. 写一个Python脚本 `scripts/strategy_dashboard.py`：读取策略库所有追踪日志，汇总输出本周策略信号更新

**产出**：
- 策略库含至少3个策略
- 1个策略追踪面板脚本（自动汇总信号）

**验证标准**：运行 `python3 scripts/strategy_dashboard.py` 输出一个格式清晰的"本周策略信号汇总"。

#### 第5个月（11月）：策略实盘模拟——纸交易验证

**任务**：在真实市场环境下追踪策略信号，但不投入真金白银。这是从"回测好看"到"实战有用"的关键一步。

**具体操作**：
1. 选1个策略（经前4个月最优的那一个），启动"纸交易追踪"
2. 每天用Claude Code自动获取该策略的信号，记录"如果按策略操作会怎样"
3. 每周做一次对比：策略信号vs你的直觉判断vs实际市场走势
4. 记录所有"策略说买但我感觉不对"的时刻——这些是你的隐性知识在说话

**产出**：
- 1份"纸交易追踪日志"（至少4周数据）
- 1份"策略vs直觉vs市场"三方对比记录

**验证标准**：你能说出这个策略在真实环境中的3个优点和3个问题（不是回测数据里的，是你追踪过程中发现的）。

#### 第6个月（12月）：策略库年终总结+2027年量化路线

**任务**：总结6个月的量化实践，产出1份"个人投资者量化实践报告"。

**产出**：
- 策略库年终总结（含所有策略的追踪记录和风险评级）
- 1个"值得在2027年继续追踪"的策略
- 1份"量化实践心得"（可作为行内分享素材）

### 1.4 突破点1验收标准

## 二、突破点2：多Agent投研协作系统

### 2.1 为什么选这个

你目前使用Claude Code更多是"单人模式"——你说一个任务，它做一件事。但Claude Code的核心能力之一是**多Agent并行**，这在你的《使用精通手册》第4章、第6章已有记载，但从未被充分利用于投资研究。

你现在有19个金融插件，每个都是专业能力单元。如果把它们编排成**一个指令触发5个Agent同时工作**的协作系统，你获得的不是"5倍速度"，而是"5种视角同时碰撞"——这正是优秀投资决策需要的多维度交叉验证。

### 2.2 目标

到2026年12月，建成一个**一句话触发式投研命令链**。你输入"分析昆仑万维是否值得继续持有"，Claude Code自动启动5个Agent并行工作，15分钟后你收到一份多维度交叉验证的投资研判报告。

### 2.3 系统架构

```
你的一句话指令："分析昆仑万维，评估是否继续持有"
        │
        ├→ Agent A: 基本面分析（equity-research插件）
        │   └ 最新财报解读 + 管理层指引 + 投资论点检查
        │
        ├→ Agent B: 估值与对比（financial-analysis + valuation-reviewer）
        │   └ DCF估值 + 可比公司分析 + 估值合理性判断
        │
        ├→ Agent C: 市场与情绪（market-researcher）
        │   └ 行业概览 + 竞品对比 + 资金流向分析
        │
        ├→ Agent D: 风险审查（对公信贷线插件）
        │   └ 经营风险评估 + 现金流压力测试 + 信用视角
        │   (这是你的绝对差异——用银行信贷的严格标准审股票)
        │
        ├→ Agent E: 大师框架对照（从Obsidian知识库读取13位大师）
        │   └ 用你最认可的2-3位大师的框架评估这只股票
        │
        └→ 汇总Agent: 合并5份输出 → 争议点标注 → 你的决策建议
```

### 2.4 路径（每月一个里程碑）

#### 第1个月（7月）：跑通第一次5-Agent并行研究

**任务**：选1只你最熟悉的持仓股，让Claude Code同时启动5个Agent，每个执行不同维度的分析。

**具体操作**：
1. 先在CLAUDE.md中建立"投研Agent模板"（Agent名称+插件分配+输出格式要求）
2. 启动并行Agent，观察：哪些Agent输出质量高？哪些跑偏了？哪些结论互相矛盾？
3. 记录并行执行的问题（Agent间通信、输出格式不一致、等待时间过长等）

**产出**：
- 1份"5-Agent并行投研报告"（含5个维度的原始分析+1份汇总）
- 1份"首次并行执行复盘"：问题清单和改进方向

**验证标准**：5个Agent都成功完成各自任务，汇总报告标注了观点一致和矛盾的地方。

#### 第2个月（8月）：Agent角色细化——引入"对手方"视角

**任务**：在5个Agent基础上增加1个关键角色——"魔鬼代言人Agent"。这个Agent的指令是：**找出所有不持有/卖出的理由**。

**为什么重要**：人类有确认偏误（只找支持自己观点的证据）。你自己的Agent团队也可能倾向于"附和"你的持仓偏好。一个专门找反方论据的Agent是最重要的理性校验机制。

**具体操作**：
1. 设计"魔鬼代言人Agent"的提示词模板
2. 在并行投研中增加第6个Agent
3. 对比"有反方"vs"无反方"的报告质量差异

**产出**：
- "魔鬼代言人Agent"提示词模板（可复用）
- 1份含反方视角的投研报告

**验证标准**：反方Agent至少提出3个你之前没想到的风险点。

#### 第3个月（9月）：建立Agent编排指令库

**任务**：不再每次都手工编排，而是建立一套标准化的Agent编排指令库，一句话触发全流程。

**具体操作**：
1. 建立3套标准编排模板：
   - **模板A（深度个股）**：6 Agent，完整投研，30-40分钟
   - **模板B（快速扫描）**：3 Agent，快速判断，10-15分钟
   - **模板C（行业月报）**：4 Agent，行业全景扫描，20-25分钟
2. 将每个模板写入一个"触发指令"（一段文字），保存在 `deliverables/ai/agent_orchestration/`
3. 验证：开新对话，粘贴指令，看是否自动触发正确的Agent配置

**产出**：
- 3套标准化编排指令
- 每个模板1次实战验证的产出

**验证标准**：新对话中粘贴指令，不需要额外解释，Claude Code自动按模板启动对应的Agent。

#### 第4个月（10月）：Agent→知识库双向流动

**任务**：让Agent的研究产出**自动沉淀回Obsidian知识库**，而不是留在对话记录里变成一次性产物。

**具体操作**：
1. 设计一个"投研落地"流程：
   ```
   Agent完成研究 → 汇总报告生成 → 
   ├→ 关键发现写入对应Obsidian笔记（如持仓追踪文件）
   ├→ 新增行业认知自动创建知识卡片
   └→ 风险提示写入"投资风险日志"
   ```
2. 写一个Python脚本 `scripts/agent_to_obsidian.py`：读取Agent汇总报告，按规则提取关键信息，写入指定的Obsidian Vault文件
3. 跑通一次"Agent研究→自动更新Obsidian"的闭环

**产出**：
- `agent_to_obsidian.py` 脚本
- 1次完整的"研究→沉淀"闭环验证

**验证标准**：运行脚本后，Obsidian中对应的笔记文件内容有更新，且更新的内容与Agent研究报告一致。

#### 第5个月（11月）：建立"投研日报"自动化

**任务**：将多Agent系统从"按需触发"升级为"定时触发"，每天早上自动生成一份投研晨报。

**具体操作**：
1. 设计晨报Agent编排（轻量版，2-3 Agent）：自选股异动 + 宏观事件 + 今日催化剂
2. 写一个脚本 `scripts/morning_research_brief.py`，功能：
   - 启动时自动触发轻量版投研Agent
   - 将输出写入 `deliverables/investment/每日投研晨报/`
3. 设置 `/loop 1d` 自动触发（或手动每天早上运行脚本）

**产出**：
- 投研晨报模板
- `morning_research_brief.py` 脚本
- 至少7天的晨报记录

**验证标准**：连续7天产出晨报，格式统一，内容不重复。

#### 第6个月（12月）：多Agent系统年终评估

**任务**：评估半年多Agent投研的效果，决定哪些Agent编排值得保留，哪些需要优化。

**产出**：
- 多Agent投研系统总结报告
- "2027年最值得优化的2个Agent"
- 1份"Agent编排最佳实践"（可作为行内/社区分享素材）

### 2.5 突破点2验收标准

## 三、突破点3：知识库AI操作系统

### 3.1 为什么选这个

你目前有32个Markdown文件的Obsidian Vault，这是你全部个人知识的物理载体。但现状是：**它是一个被动仓库，不是一个主动协同者**。

类比：你的Vault现在是图书馆——书都分门别类放好了，但你需要自己去翻找。你要的是一个"研究助理"——当你开始思考一个问题时，它主动把相关的知识卡片、大师观点、过往笔记、跨域框架推到你面前。

这个升级需要两部分配合：
- **Obsidian端**：结构化知识卡片、MOC体系、双向链接
- **Claude Code端**：读取Vault内容，在你研究时自动关联相关知识

### 3.2 目标

到2026年12月，Obsidian Vault从"被动的Markdown仓库"升级为"Claude Code可以读取、理解、关联的AI可用知识系统"。当你和Claude Code讨论一个投资问题时，它能自动调取你的大师卡片、过往研判、跨域框架作为上下文。

### 3.3 系统设计

```
你："这只科技股现金流很差，但成长性高，值不值得投？"
        │
Claude Code 自动读取你的知识库：
        │
        ├→ 大师卡片：费雪（成长股投资）→ 15个要点中"利润率"是关键
        ├→ 大师卡片：格雷厄姆（安全边际）→ "现金流差=没有安全边际"
        ├→ 你的过往笔记：3个月前你说"高成长低现金流的公司A股要打折看"
        ├→ 跨域框架：Self-Evolution → "企业不同阶段的现金流分布规律"
        └→ 你的风险偏好（从投资框架提取）："永远不上杠杆"
        │
        └→ 综合输出：基于你全部知识体系的投资建议
```

### 3.4 知识库现状诊断

当前Obsidian Vault的问题：

| 问题 | 现状 | 影响 |
|------|------|------|
| **知识卡片格式不统一** | 13位大师卡片长度不一（50-70行），格式各写各的 | Claude Code难以结构化理解 |
| **MOC停留在目录层面** | 8个MOC主要是文件列表，缺少"这个领域的核心命题是什么" | 无法支撑深度检索 |
| **跨域连接是手动维护的** | 靠你手动添加 `[[双向链接]]`，没有自动发现机制 | 连接数量取决于你当天记不记得 |
| **与Claude Code的联动是单向的** | Claude Code可以读到Obsidian文件，但不会"主动去关联" | 知识库只是文件，不是AI可用的知识图谱 |
| **缺乏"增量学习"机制** | 每次研究产生的新认知留在了对话记录里，没有回流到知识库 | 知识库的增长率低于你的实际学习速度 |

## 四、Claude Code成为"第二大脑"的协同架构

### 4.1 现状：Claude Code目前是什么

当前Claude Code在你工作流中的角色是 **高级研究助理+编程助手**：

- 你给它任务，它执行
- 你能用记忆系统让它记住偏好
- 你能用19个插件扩展它的金融分析能力
- 但它还不是"协同者"——它不会主动发现你需要什么、不会自动关联你的知识库、不会在你没有明确指令时提供有价值的信息

### 4.2 目标：2026年底Claude Code要成为什么

```
┌────────────────────────────────────────────┐
│              Claude Code 第二大脑           │
│                                            │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐ │
│  │ 知识层    │  │ 研究层    │  │ 执行层    │ │
│  │          │  │          │  │          │ │
│  │ 读取     │  │ 多Agent  │  │ 生成报告  │ │
│  │ Obsidian │→│ 并行研究  │→│ 创建PDF  │ │
│  │ 关联     │  │ 19插件   │  │ 更新文件  │ │
│  │ 知识卡片  │  │ 大师框架  │  │ 运行脚本  │ │
│  └──────────┘  └──────────┘  └──────────┘ │
│                                            │
│  ┌──────────────────────────────────────┐  │
│  │ 记忆层：CLAUDE.md + auto-memory       │  │
│  │ 你的身份、偏好、目标、过往决策        │  │
│  └──────────────────────────────────────┘  │
└────────────────────────────────────────────┘
```

关键是中间那一层——Claude Code不是在"知识"和"执行"之间做一个被动的搬运工，而是在**研究层**主动做多维度分析。

### 4.3 三个协同场景（2026年底要达到的能力）

#### 场景一：投资决策协同

**你说的**："我关注的那只半导体设备股今天跌了5%，要不要加仓？"

**Claude Code自动做的**：
1. 读取你的知识库：大师卡片（巴菲特/费雪/霍华德·马克斯的买入时机原则）、你的投资框架（再平衡规则、行业配置上限）
2. 启动并行Agent：
   - Agent A：/earnings + /model-update 检查最新基本面变化
   - Agent B：/competitive-analysis 检查行业竞品是否同步下跌（系统性风险 vs 个股风险）
   - Agent C：读取你的持仓CSV，计算加仓后的行业集中度变化
3. 输出：不是"买"或"不买"，而是"根据你的持仓结构和投资原则，当前情况对应以下三个问题需要你判断..."
4. 等你决策后，自动更新你的持仓追踪文件和投资日志

#### 场景二：工作提效协同

**你说的**："下周要去一家AI芯片企业做现场尽调，帮我准备。"

**Claude Code自动做的**：
1. 读取知识库：科技金融框架、你对半导体行业的过往研判、你的尽调提示词模板
2. 启动Agent：
   - Agent A：/meeting-prep-agent 生成拜访简报（企业背景+关键人物+财务摘要）
   - Agent B：用对公信贷线插件跑一遍企业公开信息（工商+司法+舆情）
   - Agent C：比对你的"尽调清单模板"，标注本次需要特别关注的点（基于企业所在细分领域——比如GPU芯片受制裁影响）
3. 输出：一份定制化尽调准备包（企业简报 + 关键问题清单 + 风险检查点）

#### 场景三：知识管理协同

**你说的**："最近读了一篇关于银行AI风控的文章，有些启发。"

**Claude Code自动做的**：
1. 问："要沉淀到知识库吗？"
2. 创建知识卡片（按标准frontmatter模板）
3. 扫描知识库已有内容，自动建议双向链接："这篇文章讲的模型可解释性要求，和你3个月前写的AI合规红线有关联。要链接吗？"
4. 更新MOC中的"最新添加"部分
5. 如果你说"帮我做一篇关于银行AI风控的知识总结"，自动聚合相关卡片+跨域框架，生成初稿

## 六、三个突破点的协同效应

三个突破点之间存在正反馈循环：

```
知识库升级（突破点3）
    ↓ 提供大师框架、行业认知、个人判断规则
多Agent投研（突破点2）
    ↓ 基于知识库内容，并行执行多维度研究
量化策略管线（突破点1）
    ↓ Agent研究结果→可回测的策略→策略回馈知识库
    ↓ 策略运行中发现的新认知→通过增量管道回流知识库
    ↓ 知识库更丰富→Agent研究更深→策略更精准
```

**关键洞察**：三个突破点分开做，每件事单独做都有价值。但合在一起做，它们互相增强：
- 没有知识库（突破点3）的Agent研究，是没有"你自己"的研究——它会做通用的金融分析，但不知道你的偏好和判断
- 没有Agent编排（突破点2）的知识库，是没有人使用的图书馆
- 没有量化管道（突破点1）的投研，是只有定性没有定量的分析——你把银行信贷的严格数字习惯带进了投资，但缺少量化工具验证你的判断
