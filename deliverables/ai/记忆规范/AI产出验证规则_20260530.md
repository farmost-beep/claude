# AI产出验证规则

> 基于治理型Agent框架 + 一致性格栅原则
>
> 不是"检查AI有没有犯错"——是"在哪些地方、用什么方式、验证AI的输出没有偏"

---

## 验证分级

| 级别 | 适用场景 | 验证方式 | 触发条件 |
|:---|:---|:---|:---|
| L0 无需验证 | 格式转换、数据搬运、已知规则检查 | 自动通过 | 操作型Agent的确定性任务 |
| L1 规则自动验证 | 格式完整性、数值范围、必填字段 | Agent自动扫描 | 每次飞轮运行 |
| L2 第二实例交叉验证 | 投资分析结论、风险评估、黑天鹅场景 | 另一个实例独立分析+比对 | 高风险分析任务 |
| L3 反向验证 | Agent的关键推理链路 | 反向推演+前提爆破 | L2发现不一致时升级 |
| L4 人工验证 | 买卖建议、策略调整、跨飞轮矛盾仲裁 | 人审查+决策 | L3无法消解的矛盾 |

---

## 每级具体规则

### L0：无需验证
- 格式转换（JSON→表格）
- 价格数据抓取（Sina API→本地）
- 文件存在性检查
- 日期/时间比对

### L1：规则自动验证
检查项（每次飞轮运行自动执行）：
- [ ] 报告是否包含所有必需章节
- [ ] 数值是否在合理范围内
- [ ] 日期/时间戳是否一致
- [ ] 引用来源是否标注
- [ ] 前一版本中的已知错误是否修复

### L2：第二实例交叉验证
触发条件（任一满足即触发）：
- 投资飞轮产出黑天鹅场景
- 持仓风险分析中出现"高风险"标记
- AI能力飞轮对某飞轮的可信度评分≤B级
- 两个飞轮产出了方向性矛盾的信号

验证方式：
1. 用不同随机种子/不同模型，独立分析同一输入
2. 比对两个实例的结论
3. 结论一致→通过，记录"双实例一致"
4. 结论不一致→升级到L3

### L3：反向验证
方式一（前提爆破）：
- 提取推理链中每个关键前提
- 逐一假设"这个前提不成立"
- 检查结论中哪些部分随之改变
- 如果改变的部分是结论的核心→标记为"前提敏感"

方式二（反向推演）：
- 给定Agent的输出
- 让另一个实例反向推演：如果这个输出是对的，原始输入应该是什么样的
- 比对推演的输入和实际输入是否一致

### L4：人工验证
场景：
- 投资建议（买卖/清仓/调仓）
- 跨飞轮矛盾无法自动消解
- Agent的行为触及灰色区域
- 验证规则本身需要修改

原则：
- 不是"AI做错了让人修"——是"AI的分析和矛盾结构化呈现，人做最终判断"
- 人的判断记录为"先例"，同类情况以后可能降级到L1/L2

## 一致性格栅

核心原则：**两个独立判断源指向一致，才能采信。只有一个判断源，无论它多强，都要打折。**

具体规则：
- L2交叉验证的两个实例必须独立——不同的随机种子，最好不同的模型提供商
- 如果两个实例一致→结论可信度+1级
- 如果两个实例不一致→不能采信任一方的结论，升级到L3或L4
- 如果只有单一判断源→结论标注"未经独立验证"

## 验证日志格式

每次验证后记录：

```
[验证ID] [日期] [飞轮/任务]
验证级别：L0/L1/L2/L3/L4
验证对象：[具体是什么产出]
验证方式：[自动扫描/第二实例/反向验证/人工]
验证结果：通过/不通过/需升级
发现：[通过则写"无异常"，不通过则写具体发现]
行动：[不通过时，采取了什么行动]
```

## 规则进化

每季度回顾：
- 哪些L1规则总是通过？→ 考虑降级到L0（减少噪音）
- 哪些L2验证总是触发L3升级？→ L2的触发条件可能需要收紧
- 哪些L3反向验证发现了真实问题？→ 这类问题的检测规则需要前置到L1
- 人工验证（L4）的决策中，有多少次推翻了AI的结论？→ 如果频率上升，L2/L3需要加强
