## 第一章：核心理念

### 1.1 Claude Code 是什么

Claude Code 是 Anthropic 公司推出的终端级 AI 编程助手。你的理解可以这样定位——它不是你"问一个问题，得到一个回答"的聊天工具，而是一个**驻留在你项目目录中的全天候协作伙伴**。

它有四个核心能力，这四点决定了它能做什么：

1. **读写文件**：直接访问你的代码库、Markdown 文档、数据文件、配置——不需要你复制粘贴
2. **执行命令**：可以运行 shell 命令、Python 脚本、git 操作——它会自己动手
3. **多 Agent 并行**：同时分出多个子 Agent 处理独立任务——一个负责投资分析，一个负责家庭规划，互不等待
4. **记忆系统**：通过 CLAUDE.md 等文件记住你的背景、偏好、目标——每次对话都不是从零开始

### 1.2 它不是什么

- **不是 ChatGPT/网页版 Claude**：ChatGPT 和 claude.ai 是无状态的对话窗口。你每次打开都是新人见面。Claude Code 记住你。
- **不是 IDE 插件**：不是侧边栏里的代码补全。它拥有完整的文件系统和命令执行权限。
- **不是替代你思考**：它不会替你下投资决策，但会帮你把13位投资大师的分析框架跑一遍，然后把结论摆在桌上让你判断。

### 1.3 正确的使用心态

把 Claude Code 当成你的**研究助理团队**，而不是一个问答机器人。

- 告诉助理团队目标（"研究昆仑万维是否值得持有"），而不是每条指令都微管理（"帮我查一下昆仑万维的PE"这类单次问答适合 ChatGPT）
- 助理团队会干活，但需要你审阅结果、做最终决策
- 越是复杂的、需要多步骤推理的任务，Claude Code 越有价值。简单的一次性问答用 ChatGPT 就行

## 第二章：对话技巧

### 2.1 写好提示词的核心原则

Claude Code 不需要精致的"提示词工程"。遵循三条就够了：

**原则一：说清楚目标，而非步骤**

坏的写法：`帮我创建一个文件，写投资自由的定义，然后创建一个PDF，然后...`

好的写法：`把投资自由路线图生成一版PDF，用中文字体，输出到 deliverables/investment/`

区别：前者是命令清单，后者是结果定义。Claude Code 自己知道怎么分解步骤。

**原则二：带上上下文**

每次对话，Claude Code 自动加载 CLAUDE.md。但你可以在对话中追加临时上下文：

- 文件路径：`参考 deliverables/investment/投资自由路线图.md 的结构`
- 运行结果：`刚才生成PDF报错了，错误信息是...`
- 你的判断：`这个分析偏乐观了，实际昆仑万维的现金流比表面差很多，重新评估`

**原则三：明确验收标准**

`如果PDF生成了，告诉我文件大小和页数`

这一句就够了。Claude Code 会自己验证结果。

### 2.2 多轮对话策略

Claude Code 最强的地方不是一次性长 Prompt，而是**多轮协作**。

典型的多轮模式：

1. 第一轮：定义方向和范围（"帮我做一个投资自由路线图，面向我自己，目标10年内实现投资自由"）
2. 第二轮：调整内容（"把'被动收入'这个定义改一下，用布正书的框架"）
3. 第三轮：生成产出（"生成PDF版本"）
4. 第四轮：迭代优化（"PDF字体字号换成STHeiti 12pt，太大会溢出"）

每轮聚焦一件事。不要试图在一个 Prompt 里说完所有要求——你不需要一开始就完美，边做边调就是正确姿势。

### 2.3 喂上下文的最佳方式

**方式一：文件路径引用（最推荐）**

Claude Code 直接读你的文件，不需要你复制粘贴内容：

`参考 deliverables/investment/投资自由路线图.md 的投资框架部分，用同样的结构分析另一只股票`

**方式二：在对话中追加关键数据**

当需要临时数据时，直接粘贴关键片段：

`这是体检报告的三项异常指标：空腹血糖6.2、总胆固醇5.8、低密度脂蛋白3.6。基于这些数据更新健康计划。`

**方式三：引用已有对话产物**

`上一轮生成的投资自由路线图里，第四阶段的投资组合配置，把港股比例从30%调到20%`

**不推荐的方式**：把所有可能相关的文件内容全部贴进 Prompt。Claude Code 能自己读文件——让它读。

## 第三章：记忆系统

### 3.1 记忆系统的架构

Claude Code 的记忆系统由两层构成：

**第一层：CLAUDE.md（项目级 + 用户级）**

核心文件是项目根目录下的 `CLAUDE.md`，它被每次对话自动加载。

在你当前的项目中，`CLAUDE.md` 保存了：
- 你的个人信息（陈颖芳，1977年生，中科大，邮储银行）
- 你的六大目标（投资、事业、家庭、健康、AI能力、知识库与思维框架）
- Karpathy 四条原则（保持简单、手术刀式改动等）

除了 CLAUDE.md，还有一个扩展记忆体系：**分类记忆文件**。你可以将特定话题的长期记忆拆分到独立文件中，例如：
- `deliverables/investment/投资自由路线图.md` —— 投资目标、框架、数据
- `deliverables/health/百岁健康计划.md` —— 健康指标、计划、进度

**第二层：Auto Memory（自动记忆系统）**

Claude Code 还具备持久化的 auto memory 系统（位于 `~/.claude/projects/`），由 `MEMORY.md` 索引和分类记忆文件组成。四种记忆类型：

| 类型 | 说明 | 你的实际记忆文件 |
|------|------|------------------|
| **user** | 关于你的固定背景信息 | `user_profile.md`（六大目标、投资理念） |
| **feedback** | 你对产出的偏好和反复出现的修改意见 | `feedback_coding_style.md`（Karpathy原则）、`feedback_verify_file_write.md`（写后验证） |
| **project** | 项目级别的规则和约定 | `goal_docs_pdf.md`（双版本维护）、`investment_masters_project.md` |
| **reference** | 外部系统指针 | 待补充（如 Linear 项目、Grafana 面板等） |

两层的分工：
- **CLAUDE.md**：每次对话自动加载，放最核心的身份+原则，保持精简
- **Auto Memory**：按需查阅，放详细规则、项目约定、反馈记录，可以更丰富

这种双层设计的好处：CLAUDE.md 不会膨胀影响对话效率，但详细规则仍然持久存在、跨对话可查。

### 3.2 四种记忆类型

在 Claude Code 的记忆体系中，可以按**用途**区分四种记忆：

| 类型 | 说明 | 例子 |
|------|------|------|
| **user** | 关于你的固定背景信息 | 出生年份、职业、学历、风险偏好 |
| **feedback** | 你对产出的偏好和反复出现的修改意见 | "PDF字体用STHeiti不要用PingFang"、"投资分析偏好保守估值" |
| **project** | 项目级别的规则和约定 | 目录结构约定、Markdown+PDF双版本规范、Karpathy原则 |
| **reference** | 可引用的知识框架 | 布正书的投资自由定义、13位大师分析模型 |

### 3.3 何时存，何时不存

**应该存入记忆的**：
- 反复出现的偏好（如字体、格式、风险偏好）
- 需要跨对话延续的背景（健康指标基准值、子女年龄和教育阶段）
- 你发现 Claude Code 反复犯同样的错，需要在 CLAUDE.md 里写规则纠正它

**不应该存入的**：
- 临时性的上下文（这只股票今天分析的结论——下一次分析可能结论不同）
- Claude Code 自己能查到的事实（公司年报数据、股票代码）
- 太琐碎的细节（会让 CLAUDE.md 膨胀，降低每次对话的效率）

**实际的判断标准**：问自己"这个信息下个月打开 Claude Code 还有用吗？"如果答案为是，存。如果答案为否，不存。

### 3.4 记忆的持久化机制

CLAUDE.md 在项目根目录，它是**项目级记忆**——换了电脑、换了终端，只要在同一个 git 仓库里，Claude Code 都能加载它。

你还可以有**用户级记忆**，位于 `~/.claude/CLAUDE.md`，在所有项目中生效。

两者的关系：Claude Code 加载用户级 CLAUDE.md，再加载项目级 CLAUDE.md。用户级放跨项目通用信息（如你是谁、基本原则），项目级放特定项目的信息（如目录结构、依赖）。

## 第四章：Agent 系统

这是 Claude Code 区别于所有其他 AI 工具的核心功能。

### 4.1 Agent vs 直接对话

**直接对话模式**：你问，Claude Code 答。一个请求一个响应。适合简单操作——改一行配置、查一个信息、生成一个短文本。

**Agent 模式**：你给目标，Claude Code 自主规划步骤、调用工具、检查结果、迭代修正，直到完成目标或遇到阻塞。适合复杂任务——生成一份完整的投资分析报告、构建一个PDF生成脚本、同时处理六个目标的文档更新。

判断标准：
- 单步骤 → 直接对话
- 多步骤 → Agent
- 需要并行处理多个独立任务 → 多 Agent

### 4.2 Subagent 类型

Claude Code 可以分派不同类型的 Subagent：

- **general-purpose**：通用 Agent，胜任大多数任务。默认类型。
- **Explore**：专门用于代码库探索，擅长在大项目中快速定位相关文件、理解代码结构。适合"帮我找到所有跟PDF生成相关的脚本"这类搜索任务。
- **Plan**：专门用于先规划再执行。适合需要先制定方案再动手的复杂任务，如"我需要重构整个PDF生成流程，先帮我列出方案"。
- **claude-code-guide**：Claude Code 自身的指南 Agent，适合回答关于 Claude Code 使用的问题。

### 4.3 并行 Agent 的正确用法

这是你已经在使用的最强功能。并行 Agent 的核心原则：

**同时处理互不依赖的独立任务。**

正例（你在项目中已实践的模式）：同时跑六个 Agent，每个负责一个目标领域（投资、事业、家庭、健康、AI、知识库）的文档更新。六个目标之间互不依赖，Agent 互相看不见对方——这正是并行的最佳场景。

反例：先让 Agent A 生成投资分析，再让 Agent B 基于那个分析生成PPT。这两个任务有依赖关系（B需要A的输出），不能并行。应该先串行执行 A，再串行执行 B，或者在A完成后自动触发B。

**并行 Agent 的汇总模式**：

并行出不等于不汇总。典型流程：
1. 同时分发 N 个 Agent，每个处理独立模块
2. 所有 Agent 完成后，你用直接对话模式（或一个新 Agent）汇总结果
3. 汇总时聚焦"交叉检查"和"一致性"——比如六个目标计划的时间线是否冲突、资源是否重复计算

### 4.4 后台运行

`run_in_background` 参数让 Agent 在后台执行，你不需要守着它。

适用场景：
- PDF 生成（你的 build_goal_pdfs.py 跑一次要处理六个文件，耗时较长）
- 需要等待外部 API 响应的任务
- 你不急于拿到结果、可以先做别的事的任务

使用方式：在给出任务指令时，加上"在后台运行"或让 Claude Code 使用 `run_in_background` 参数。

### 4.5 何时用 Agent，何时自己动手

**用 Agent**：
- 需要多步骤、多工具协作的复杂任务
- 需要并行处理多个模块
- 你不确定具体怎么做的探索性任务（让 Agent 尝试，你审阅结果）

**自己动手（直接对话）**：
- 单文件修改（改一行配置、修一个 bug）
- 快速信息查询（"这个文件有多大"、"git log 最近5条"）
- 你很清楚要做什么，只是需要 Claude Code 执行

**一个实用原则**：如果你能用一句话说清楚要做什么，就直接对话。如果需要先计划再执行，就用 Agent。

## 第五章：工具链精通

### 5.1 文件操作：Read / Write / Edit

| 工具 | 用途 | 何时用 |
|------|------|--------|
| **Read** | 读取文件内容 | 任何需要知道现有内容的场景。Claude Code 默认会先读文件再编辑 |
| **Write** | 创建新文件或完全重写 | 新建文件，或者改动太大、Edit 不如重写方便时 |
| **Edit** | 精确替换文件中的某段文本 | 小范围修改。只改动需要改的地方，不改格式、不改相邻代码 |

**关键教训**：Claude Code 的 Write/Edit 是**真实写入文件系统的**。但 Agent 模式下的写入行为有一个值得注意的陷阱——详见第八章 8.1 节。

### 5.2 命令执行：Bash

Claude Code 可以执行几乎任何终端命令。你最常用的几类：

- **python3 脚本执行**：你的所有 build_*.py 脚本都通过 Bash 工具运行
- **git 操作**：`git status`、`git diff`、`git commit`（注意：Claude Code 不会主动提交，除非你明确要求）
- **文件系统操作**：`ls`、`find`、`mkdir`
- **python3 -c**：快速验证（如检查库是否安装：`python3 -c "from fpdf import FPDF"`）

**权限控制**：你的 `.claude/settings.local.json` 中已经配置了常用命令的自动放行（允许 git 命令、python3 脚本等）。这是减少每次确认弹窗的关键配置。

### 5.3 任务跟踪：TaskCreate / TaskUpdate

在 Agent 模式下，Claude Code 可以创建和更新任务。对于你自己，你不需要直接调用这些——它们是 Claude Code 内部使用的。但**你需要知道它们存在**，因为当你审查 Agent 行为时，任务列表会让你看清 Agent 在做什么、做到哪一步了。

### 5.4 网络搜索：WebSearch / WebFetch

- **WebSearch**：搜索互联网信息。适合查最新数据、行业新闻、政策变化。
- **WebFetch**：抓取指定 URL 的内容。适合读取公开的文档、API 文档、研究报告。

你的权限配置中，WebSearch 已全局放行，WebFetch 只放行了 `raw.githubusercontent.com` 域。这是合理的安全策略——不要对任意 URL 放行 WebFetch。

### 5.5 Skill 技能系统

Skill 是 Claude Code 的"专业模块"。当前你的项目环境中可用的 Skill：

| Skill | 用途 | 何时触发 |
|-------|------|----------|
| **update-config** | 修改 Claude Code 配置（settings.json） | 新增权限放行、修改环境变量 |
| **verify** | 验证代码变更是否正确 | 改完脚本后跑一遍确认效果 |
| **run** | 启动和运行项目 | 让 Claude Code 帮你运行应用 |
| **fewer-permission-prompts** | 减少权限弹窗 | 让你常用的命令自动放行 |
| **code-review** | 代码审查 | 提交前检查代码质量 |

你对 Skill 不需要特别学习——你只需要知道：当你对 Claude Code 说"帮我把 python3 命令加到自动放行列表"，Claude Code 会自动调用 update-config Skill 来完成。

## 第六章：实战工作流

### 6.1 六大目标的 Claude Code 使用模式

基于你已有的项目结构，六个目标各自有不同的最适配的 Claude Code 模式：

**投资**：
- 多 Agent 并行分析多只股票（每个 Agent 独立跑一只股票的大师模型）
- 13 位大师分析框架作为 reference 记忆存储
- PDF 报告自动生成（已有成熟的 build_goal_pdfs.py 链路）
- 适合定期（每周/每月）跑一次更新

**事业**：
- 演讲/汇报材料快速生成（PPT、演讲稿）
- 科技金融框架文档的迭代完善
- 适合"有明确产出"的任务驱动模式

**家庭**：
- 子女教育规划的时间线推演
- 家庭事务清单生成
- 适合长期规划类文档，变动频率低

**健康**：
- 体检报告数据解析与对照分析
- 百岁健康计划的定期进度更新
- 适合年度/季度更新模式

**AI 能力提升**：
- 这份手册本身就是 AI 产出
- 后续适合"学习笔记"模式——每次学到新用法，追加到对应章节

**知识库与思维框架**：
- Obsidian Vault `6goals` 的内容维护（MOC 页面、知识卡片、wikilink）
- 跨域连接发掘（如 INV↔TF 的双向链接建立）
- 适合"每周维护"模式——每周日 30 分钟批量更新

### 6.2 多 Agent 并行工作流

**何时分发**：
- 任务之间零依赖（六大目标的文档更新互不依赖 → 可并行）
- 每个任务的数据源独立（研究A股不需要等港股的数据 → 可并行）
- 每个任务有清晰的独立验收标准

**分发前的准备**：
1. 确保每个 Agent 能访问到它需要的文件（目录已存在、权限已配置）
2. 给每个 Agent 一个明确的目标和验收标准
3. 用 CLAUDE.md 里的 Karpathy 原则约束 Agent 行为（保持简单、手术刀式改动）

**如何汇总**：
1. 所有 Agent 完成后，打开一个新对话
2. 逐个检查各 Agent 的输出文件
3. 对不一致的地方提出调整指令
4. 如果需要交叉验证（如多个投资分析的结论矛盾），在汇总对话中处理

### 6.3 Markdown + PDF 双版本维护流

你已经在用的模式：

```
Markdown 源文件 → build_goal_pdfs.py → PDF 输出
```

维护规则：
1. **Markdown 是唯一编辑对象**。PDF 永远从 Markdown 生成，不要手动编辑 PDF
2. **改内容只改 Markdown**。改完后运行 build_goal_pdfs.py 重新生成 PDF
3. **Markdown + PDF 都在 deliverables 目录下**——方便 git 追踪历史版本
4. **生成脚本单独维护**（`scripts/build_goal_pdfs.py`），不跟内容耦合

每次改完 Markdown 后，用一句指令完成 PDF 更新：

```
运行 scripts/build_goal_pdfs.py 生成最新PDF
```

### 6.4 记忆更新流

你的记忆文件分布在三个位置：

| 位置 | 内容 | 更新频率 |
|------|------|----------|
| `CLAUDE.md` | 个人信息、原则 | 低，基本不变 |
| `~/.claude/projects/.../memory/` | Auto Memory（用户档案、反馈、项目规则） | 中，有新规则或偏好变化时更新 |
| `deliverables/*/` 各目标文档 | 目标定义、进度 | 中，按需更新 |

更新原则：
- 你发现新偏好 → 更新 Auto Memory 文件 + CLAUDE.md（如"写文件后必须验证"）
- 目标有了里程碑式的变化 → 更新对应 deliverable 文档
- 日常的小进度 → 不需要每次记录，积累到一定量再批量更新

### 6.5 Obsidian Vault 与知识库联动

你在 `/Users/cyingfang/Documents/Obsidian Vault 6goals/` 建立了目标导向的 Obsidian Vault：

## 第七章：进阶技巧

### 7.1 大文件处理策略

你的健康目标文档有 55KB、投资文档有 30KB，这些在 Claude Code 里都是小文件。但如果将来遇到真正的大文件（几 MB 的 Markdown、几十页的 PDF），策略如下：

- **按节读取**：用 Read 工具的 offset/limit 参数只读需要的部分，不要一次加载全文件
- **拆分存储**：大文档拆成多个小文件。如投资自由路线图的各个阶段独立成文件
- **用脚本处理**：对于超大数据（如股票历史行情），写一个 Python 脚本预处理，Claude Code 只看处理结果

当前你的文件规模还不构成问题。

### 7.2 错误恢复与重试

Claude Code 在执行中遇到错误时的行为：
- 如果它能自己解决（如文件权限问题），它会自动尝试修复
- 如果它不能（如缺少依赖库），它会停下来向你报告

你的应对策略：
- **依赖缺失**：告诉 Claude Code 安装缺失的库（`pip install`）
- **逻辑错误**：描述"期望行为 vs 实际行为"的差异，让 Claude Code 修正
- **Agent 跑偏**：如果 Agent 做的不是你想要的，不要试图纠正它。直接结束当前对话，开新对话重新描述需求。试图在跑偏的上下文里纠正比重新开始更难。

### 7.3 上下文管理：何时开新对话

Claude Code 的上下文窗口很大，但**不意味着你应该一直不换对话**。

以下信号提示你该开新对话了：
- 对话已经跨越了多个不相关的任务（投资分析 → 家庭规划 → 写邮件 → 查 bug）——上下文乱了
- Claude Code 开始引用很早之前的对话内容来回答当前问题——它在混淆上下文
- 你发现自己的 Prompt 越来越长，试图"拉回"Claude Code 的注意力——该重新开始了

**一个好的习惯**：一个对话干一件事。投资分析单开一个对话，家庭规划单开一个对话。对话之间的记忆通过 CLAUDE.md 和 deliverable 文件传递，不依赖 Claude Code 的上下文窗口。

### 7.4 Hook 与自动化

Claude Code 支持 Hook 机制——在特定时机自动执行操作。通过 `/update-config` Skill 可以配置：

- 权限放行：让你常用的命令不再弹出确认（你的 settings.local.json 已经在做这件事）
- 环境变量：设置项目级别的环境变量
- 自动化行为：如"每次对话开始前检查 git 状态"

当前你的配置已经满足了基本需求。将来如果需要更复杂的自动检查（如"每次生成 PDF 后自动校验文件大小"），可以扩展 Hook 配置。

## 第八章：常见问题与陷阱

### 8.1 Agent 没有真的写入文件

**症状**：你让 Agent 写一个文件，对话里它说"写好了"，但实际上文件不存在或内容为空。

**原因**：Agent 可能在沙盒模式或预览模式下运行，它"模拟"了写入操作但没有实际执行。也可能是 Agent 构思了内容但在需要调用 Write 工具时转而向你描述内容，而不是真正写文件。

**如何发现**：每次让 Agent 生成文件后，立即用 `ls -la` 或 Read 工具验证文件是否存在、大小是否合理。

**如何预防**：
- 对 Agent 说"写入后请用 ls 确认文件存在"
- 在 CLAUDE.md 里加入"每次写文件后必须验证文件存在"的规则
- 如果是生成脚本（如 build_goal_pdfs.py），写完脚本后运行它，看是否真的生成了输出文件

**你的实际教训**：build_goal_pdfs.py 的开发过程中，Agent 可能在某个中间对话里声称写入了修改，但实际上没有最终落地到文件系统。这就是为什么后来的修复采用的是写完脚本 → 运行脚本 → 检查输出的三步验证模式。

### 8.2 并行 Agent 的依赖问题

**症状**：你同时启动两个 Agent，A 应该生成数据，B 应该基于 A 的输出做分析。但 B 在 A 完成前就开始了，用到的数据是旧的。

**解决方案**：并行 Agent 必须用于**无依赖**的任务。如果有依赖，必须串行。判断依赖的标准很简单——问自己："B 的任务描述里，有没有提到 A 的输出？"如果有，不能并行。

### 8.3 记忆过时

**症状**：Claude Code 引用了你半年前的健康指标来分析当下的体检报告。或者固守一个你早已废弃的投资策略。

**原因**：CLAUDE.md 和 deliverable 文档没有及时更新。

**解决方案**：定期检查。每季度至少一次，打开 CLAUDE.md 和各目标文档，确认信息是否还准确。你已经在健康计划里做了进度更新——把这个习惯延伸到记忆文件。

### 8.4 权限与安全

你的 settings.local.json 已经配置了命令白名单。但需要注意：

- **不要对危险命令自动放行**：如 `rm -rf`、`sudo`、`git push --force` 这类，应该始终要求确认
- **不要对所有 URL 放行 WebFetch**：你当前只放行了 raw.githubusercontent.com，这个策略应保持
- **项目级 vs 用户级设置**：settings.local.json 是项目级的，`~/.claude/settings.json` 是全局的。敏感配置（如环境变量）放在项目级

### 8.5 Claude Code 会"偷懒"

**症状**：你让 Claude Code 写一个完整的分析报告，它只给了概要，然后建议你"自己补充细节"。

**原因**：Claude Code 在权衡"工作量 vs 价值"时，有时会走捷径。尤其是任务描述模糊时。

**解决方案**：明确你说的是"生成完整报告"而不是"给我建议"。你的 Karpathy 第四条原则"目标驱动"——每个改动要能验证——就是对付偷懒的最好工具。再加一句："如果报告少于 3000 字，视为未完成。"

### 8.6 多个文件源的一致性

**症状**：同一个信息出现在了 deliverable、Obsidian Vault、Auto Memory 三个地方，但内容不一致。比如投资目标金额在 deliverable 里是 937 万，在 Obsidian 里是旧的 900 万。

**原因**：你的信息现在分布在三个层级：deliverable（权威源）、Obsidian Vault（导航层）、Auto Memory（记忆层）。更新一个地方时容易遗漏其他位置。

**解决方案**：
- **deliverable 是唯一权威源**：任何涉及目标数据、里程碑、计划的更新，只改 deliverable Markdown
- **Obsidian Vault 放摘要**：Vault 中的目标主页只写摘要和关键数字，通过 wikilink 引回 deliverable
- **Auto Memory 不放具体数据**：Memory 文件只放规则、偏好、约定，不放目标数据（如投资金额、体重目标）
- **一致性检查**：每季度用一句指令做交叉验证——"对比所有目标文档的 deliverable 版本和 Obsidian 版本，找出不一致的数据"
