## 一、建立认知

### 1.1 插件是什么

插件是 Claude Code 的"专业工具箱"——每安装一个插件，Claude 就多掌握一套专业技能。不像 ChatGPT 的插件需要手动选择，Claude Code 的插件**安装后自动生效**，你说中文描述需求，Claude 自动判断该调哪个插件。

```
你说："帮我分析昆仑万维的最新财报"
     ↓
Claude 自动调用 equity-research 插件的 /earnings 技能
     ↓
输出：营收变化、利润趋势、管理层指引、风险提示
```

### 1.2 三句话讲清楚整个插件体系

1. **19个插件分两条线**：投资研究线（A区，10个）处理"投股票/做研究/写报告"，对公信贷线（B区，9个）处理"放贷款/审财报/管台账"
2. **三种调用方式**：斜杠命令（`/earnings 昆仑万维`）、自然语言（"分析昆仑万维财报"）、Agent 自动编排（说"生成一份昆仑万维的完整投资报告"，Claude 自动串联多个插件）
3. **输出即成品**：所有产出可直接导出为 .xlsx / .pptx / .docx，不需要二次加工

### 1.3 和Jenkins/CI/CD工具的类比

如果你用过 Jenkins 的插件生态，理解起来很快：

| 概念 | Jenkins | Claude Code 金融插件 |
|------|---------|---------------------|
| 插件本质 | 封装好的自动化脚本 | 封装好的金融专业能力 |
| 触发方式 | Pipeline 配置触发 | 自然语言/斜杠命令触发 |
| 组合使用 | Pipeline 里串多个插件 | 一个工作流自动串联多个技能 |
| 输出 | 构建产物 | .xlsx / .pptx / .docx 成品文件 |

### 1.4 哪些能直接用，哪些需要企业账号

| 类型 | 数量 | 需要什么 | 代表 |
|------|------|---------|------|
| **技能型插件**（斜杠命令） | 10个 | 安装即用，零配置 | equity-research, private-equity, meeting-prep-agent |
| **Agent型插件**（自动编排） | 5个 | 安装即用 | market-researcher, pitch-agent, kyc-screener |
| **MCP数据源**（外部数据） | 4个 | 需要企业订阅账号 | FactSet, Moody's, S&P Global, LSEG |
| **垂直工具型** | 3个 | 安装即用（部分功能需数据源） | financial-analysis, operations, fund-admin |

> 核心工作流（投资分析、对公信贷）用技能型+Agent型即可跑通。MCP数据源是锦上添花——有企业账号时数据更精准，没有时限用公开数据（新浪财经、Wind公开数据等）替代。

**检查点**：你应该能回答——①插件是干什么的 ②19个插件分哪两条线 ③哪些插件你可以直接开始用。

## 二、组织概念

### 2.0 插件全景：19个插件按工作场景编排

不按字母排，按你**实际工作流**排——从上往下一行是一个工作步骤：

```
你的工作目标
    │
    ├─ 投资成功（持仓分析+还本计划）
    │   ├─ 晨间检查：equity-research → /morning-note + /catalysts
    │   ├─ 深度分析：financial-analysis → /dcf + /comps
    │   ├─ 行业研究：market-researcher → sector-overview
    │   ├─ 组合管理：wealth-management → /rebalance + /financial-plan
    │   └─ 报告生成：pitch-agent → pitch + ib-check-deck
    │
    ├─ 事业进阶（科技金融+对公信贷）
    │   ├─ 客户拜访：meeting-prep-agent → 拜访简报
    │   ├─ 客户准入：kyc-screener + operations → 合规筛查
    │   ├─ 贷中审查：statement-auditor + sp-global + lseg
    │   ├─ 并购模型：investment-banking + private-equity + valuation-reviewer
    │   ├─ 放款定价：model-builder + lseg
    │   └─ 贷后管理：fund-admin + gl-reconciler + month-end-closer
    │
    └─ AI能力（报告制作+知识库）
        ├─ 全套路演：pitch-agent → 自动PPT
        ├─ 财报追踪：earnings-reviewer → 财报前后更新
        └─ 知识卡片：market-researcher → 行业/竞品分析
```

### 2.1 A区：投资研究线（10插件）

#### 2.1.1 equity-research — 股票研究的"瑞士军刀"

**一句话**：覆盖从晨报到深度报告的全流程股票研究。

| 高频技能 | 做什么 | 什么时候用 |
|---------|--------|-----------|
| `/morning-note` | 自选股异动+宏观事件晨报 | 每天开盘前（2分钟） |
| `/catalysts` | 未来30天催化剂日历 | 每周一查（1分钟） |
| `/thesis 标的` | 检查投资论点是否仍然成立 | 每次想操作前（3分钟） |
| `/earnings 标的` | 最新财报分析：营收/利润/指引/风险 | 每次季报发布后（10分钟） |
| `/initiate 标的` | 首次覆盖报告（完整版） | 开新仓前 |
| `/screen` | 按PE/ROE/行业/市值筛选标的 | 寻找新机会时 |

**典型场景**：早上8:50，开盘前快速扫一眼 —— `/morning-note` 看自选股有无异动，`/catalysts` 确认今天没有重大解禁或宏观事件。

#### 2.1.2 financial-analysis — 核心财务建模

**一句话**：DCF估值、可比公司分析、三表模型——投行级的财务计算引擎。

| 高频技能 | 做什么 | 输入 |
|---------|--------|------|
| `/dcf 标的` | 自由现金流折现估值 | 股票名称或代码 |
| `/comps 标的` | 可比公司PE/PS/EV-EBITDA对比 | 股票名称 |
| `/lbo 标的` | 杠杆收购模型 | 股票名称+杠杆假设 |
| `/3-statement-model` | 三表预测模型 | 历史财务数据 |
| `/debug-model` | 自动审查模型公式错误 | 模型文件 |

**和 equity-research 怎么分工**：equity-research 告诉你"这只股票好不好"，financial-analysis 告诉你"这只股票值多少钱"。通常先用 equity-research 做定性，再用 financial-analysis 做定量。

#### 2.1.3 market-researcher — 行业研究助手

**一句话**：自动生成行业概览、竞品分析、选题报告——代替半天的人工桌面研究。

**用法**：直接用自然语言说"帮我做一份中国科技金融行业全景分析"，或"对比昆仑万维和百度在AI大模型上的竞争力"。Agent 自动搜索、整理、结构化输出。

**典型产出**：行业概览（赛道规模+增长+格局）、竞品雷达图（5-8维度打分）、选题备忘录（3-5个可深入方向）

#### 2.1.4 wealth-management — 个人财富管理

**一句话**：组合再平衡、财务规划、税务优化——适用于持仓管理和还本计划执行。

| 技能 | 场景 |
|------|------|
| `/rebalance` | 当前持仓偏离目标配置时，给出调仓方案 |
| `/financial-plan` | 输入目标（如"9月还本60万"），输出最优减持路径 |
| `/client-report` | 生成资产配置报告（可用作家庭沟通材料） |
| `/tlh` | 税务亏损收割——择机卖出亏损仓位抵税 |

#### 2.1.5 投行交易三件套（按需使用）

| 插件 | 定位 | 什么时候用 |
|------|------|-----------|
| `investment-banking` | CIM/Teaser/并购模型/买家名单 | 写民建社情民意（并购贷款政策）、做产业研究 |
| `private-equity` | 投委会备忘录/尽调清单/交易筛选 | 评估科创企业融资可行性 |
| `valuation-reviewer` | 估值方法审查/组合监控 | 验证标的估值是否合理 |

#### 2.1.6 产出自动化三件套

| 插件 | 定位 | 输入 → 输出 |
|------|------|------------|
| `pitch-agent` | 全套路演资料自动生成 | 说一个主题 → .pptx + 行业概览 + 财务模型 |
| `model-builder` | Excel财务模型自动构建 | 股票代码 → .xlsx 三表/DCF/LBO |
| `earnings-reviewer` | 财报发布后自动更新 | 财报发布 → 模型更新 + 晨报草稿 |

### 2.2 B区：对公信贷线（9插件）

> 按对公信贷全流程排列：贷前 → 贷中 → 贷后

#### 贷前：客户拜访与准入

| 插件 | 定位 | 输入 → 输出 | 耗时 |
|------|------|------------|------|
| `meeting-prep-agent` | 拜访简报自动生成 | "准备拜访XX公司" → 公司概况/合作历史/谈判策略/风险评估 | 5min |
| `kyc-screener` | 尽调材料解析+合规筛查 | 上传尽调文档 → 规则引擎评估+缺口标记 | 3-10min |
| `operations` | KYC文档批量解析 | 批量上传 → 规则网格评估 | 视文件量 |

#### 贷中：财报审核与定价

| 插件 | 定位 | 输入 → 输出 | 耗时 |
|------|------|------------|------|
| `statement-auditor` | 财报准确性/完整性审查 | 上传标的财报 → 审计就绪度报告 | 10min |
| `sp-global` | S&P信用评级+财务摘要 | "拉XX公司S&P快览" | 2min |
| `lseg` | 债券定价/收益率曲线/利率分析 | "测算当前LPR下的贷款定价区间" | 5min |
| `model-builder` | 三表模型/DCF/偿债能力 | 股票或公司名称 | 15min |
| `valuation-reviewer` | 标的估值审查 | "审查这个标的的估值合理性" | 10min |

#### 贷后：台账与报告

| 插件 | 定位 | 输入 → 输出 | 耗时 |
|------|------|------------|------|
| `fund-admin` | 贷款台账核对/利息验证/NAV验算 | 台账数据 | 视数据量 |
| `gl-reconciler` | 账务差异追踪+根因分析 | 差异数据 | 5min |
| `month-end-closer` | 月末关账全套报告 | — | 10min |

### 2.3 六大核心工作流

#### 工作流1：晨间投资检查（5分钟）

```
/morning-note          → 自选股异动 + 宏观事件
/catalysts             → 未来一周催化剂日历
如有异动 → /thesis 标的 → 判断是否操作
```

#### 工作流2：深度分析一只股票（30分钟）

```
/earnings 标的 → /comps 标的 → /dcf 标的 → /sector 行业 → /competitive-analysis 标的
→ 汇总 → /initiate 标的（生成完整覆盖报告）
```

#### 工作流3：对公信贷全流程（贷前→贷后，60分钟）

```
meeting-prep-agent → 拜访简报
kyc-screener + operations → 合规筛查
statement-auditor → 财报审核
sp-global + lseg → 信用评级 + 定价基准
model-builder → 偿债能力模型
→ 汇总为内部授信报告
```

#### 工作流4：还本计划推演（15分钟）

```
/financial-plan → 输入：持仓+现金流+目标 → 输出：最优减持路径
/rebalance → 调仓到目标结构
/client-report → 生成资产配置报告
```

#### 工作流5：民建社情民意 / 课题报告（60分钟）

```
/screen-deal → 筛科创并购案例
/merger-model → 构建并购融资模型
/dd-checklist → 生成尽调清单
/ic-memo → 汇总投委会备忘录
/pitch-agent → 生成配套PPT
/ib-check-deck → 质量审查
```

#### 工作流6：科技金融培训课件（45分钟）

```
market-researcher → 行业全景图 + 竞品雷达图
pitch-agent → 全套路演资料
deck-refresh → 更新数据
ib-check-deck → 质量审查
```

**检查点**：你应该能——①说出你的主要工作场景对应的3-4个核心插件 ②默写出一条完整工作流的步骤。

## 三、质量保障

### 3.1 三个最容易出错的坑

| 坑 | 症状 | 正确做法 |
|----|------|---------|
| **数据源混淆** | 用MCP的FactSet数据做了分析，但没确认企业订阅是否生效，分析结果的现金流数据可能有偏差 | 无订阅时明确告知Claude"用公开数据源"，或先用 sp-global 拉一把确认数据可用性 |
| **输出信任过度** | DCF估值出来"刚好"符合预期就用了，没检查关键假设（永续增长率、WACC）是否合理 | 每次问一句"/debug-model 检查这个DCF的假设"，或手动检查WACC和永续增长率 |
| **插件技能记错归属** | 想用 `/merger-model` 但调了 `financial-analysis`，实际它在 `investment-banking` 里 | 不确定时直接用自然语言说需求，让Claude自动路由——比记命令更可靠 |

### 3.2 输出质量自查清单（每次重要产出前过一遍）

| # | 检查项 | 通过标准 |
|---|--------|---------|
| 1 | 数据来源 | 知道每条数据的出处（插件/公开/企业订阅），不用不知道来源的数据 |
| 2 | 关键假设 | DCF的WACC、增长率；可比分析的peer group——假设是否合理 |
| 3 | 时效性 | 用的是最新财报还是上季度？数据日期是否标注 |
| 4 | 一致性 | 同一份报告里的估值数字是否前后一致（DCF和comps不能差3倍还同时呈现） |
| 5 | 风险提示 | 产出的分析有没有包含"这个结论在什么条件下会错" |

### 3.3 插件间的"交叉验证"模式

单独一个插件的输出可能有盲区，用两个插件交叉验证：

| 分析目的 | 主插件 | 验证插件 | 怎么验证 |
|---------|--------|---------|---------|
| 估值 | financial-analysis /dcf | financial-analysis /comps | DCF估值和可比估值差距>30%？找到原因再采信 |
| 公司质量 | equity-research /earnings | statement-auditor | 财报分析说"健康"，审计工具说"有异常"？优先信审计 |
| 行业判断 | market-researcher | sp-global | 行业报告和市场数据方向一致吗？ |
| 标的筛选 | equity-research /screen | private-equity /screen-deal | 两个筛出来的标的交集，才是高信心标的 |

### 3.4 安全红线

1. **不把插件输出当最终决策**——插件是"加速器"，不是"替代判断"。每份DCF/报告都要人过一眼关键假设
2. **不把内部数据输入到没有企业订阅的MCP**——MCP数据源传输过程中可能经第三方
3. **PPT/文档导出后检查一遍**——pitch-agent生成的内容可能存在格式微调需求
4. **客户信息的脱敏处理**——使用kyc-screener/meeting-prep-agent前，确认客户数据已脱敏或在使用合规环境中

**检查点**：你应该能——①说出三个最容易出错的坑 ②每次重要产出前自动过一遍自查清单。

## 四、持续进化

### 4.1 你的插件能力水平——四个阶段

| 阶段 | 特征 | 标志性行为 |
|------|------|-----------|
| **入门** | 知道有哪些插件，偶尔用 `/morning-note` | 每天开Claude但只聊天，不调插件 |
| **熟练** | 会用核心工作流（晨检/深度分析/还本推演） | 不用翻手册也能默写出高频命令 |
| **精通** | 能组合插件、交叉验证、优化工作流 | 自己设计新的工作流，工作产出提速50%+ |
| **专家** | 能教别人用、能给新场景匹配插件 | 帮助团队建立插件使用标准 |

**你现在在哪一层？** 如果"晨间投资检查"还没形成习惯 → 入门。如果六大工作流跑过至少3个 → 熟练。如果给同事演示过插件 → 精通。

### 4.2 每周10分钟的插件能力提升计划

| 周次 | 动作 | 目标 |
|------|------|------|
| 第1周 | 每天跑一次"晨间投资检查"工作流 | 把 `/morning-note` + `/catalysts` 变成肌肉记忆 |
| 第2周 | 拿一只持仓股跑完"深度分析"全流程 | 体验从 `/earnings` → `/dcf` → `/initiate` 的完整链路 |
| 第3周 | 尝试一个对公信贷场景（如用meeting-prep-agent生成一份拜访简报） | 打开B区插件 |
| 第4周 | 用两个插件交叉验证一次（如DCF + comps） | 建立质量保障习惯 |
| 第5-8周 | 自主设计一个工作流，跑通并记录耗时 | 进入精通阶段 |

### 4.3 性能监控——追踪你的插件效率

| 指标 | 怎么追踪 | 目标值 |
|------|---------|--------|
| 晨间检查耗时 | 从打开Claude到看完/morning-note | < 3分钟 |
| 深度分析一只股票 | 从/earnings到/initiate完成 | < 30分钟 |
| 对公信贷全流程 | meeting-prep到内部授信报告 | < 60分钟 |
| 插件命中率 | 直接用命令成功 vs 需要翻手册 | > 80% |

### 4.4 扩展方向

当你六个工作流都跑熟了，可以考虑：

- **自定义工作流**：把多个命令串成一个别名（如 `morning-routine` = `/morning-note` + `/catalysts`）
- **向团队输出**：整理一份"我们团队最常用的5个插件+3个工作流"，帮助团队提效
- **新插件探索**：关注 Anthropic Financial Services 的插件更新，新插件发布后第一时间测试
- **交叉能力**：尝试把A区（投资）和B区（信贷）的能力交叉——比如用 `model-builder` 给一个科创企业做偿债能力分析，再用 `ic-memo` 输出投委会备忘录
