## 🌱 一、起点：一个典型的"AI门外汉"

2026年4月，我的AI使用方式是：

- 偶尔打开ChatGPT，问几个问题，得到一些"听起来不错"的回答
- AI产出的东西，要么不敢直接用（怕错），要么直接用（后来发现真的错了）
- 听说有人"用AI一天干完三天的活"，我觉得要么是吹牛，要么跟我无关
- 我的工作流和AI是两条平行线——工作归工作，AI归AI

一句话总结：**我有AI工具，但没有AI能力。**

5月初，我偶然读到一份基于逆向分析Anthropic（Claude的母公司）内部工程实践总结出的《AI方法论完整手册》。18条原则，四层框架，从"人与AI的关系"到"AI自进化"。第一反应是——这也太复杂了？

但手册的第一页只有三句话：

> 1. 给AI完整上下文+权限
> 2. 给目标不给步骤
> 3. 生成后必自查，关键产出交叉审查

"就从这三条开始"，我对自己说。

5月30日，我花了5个小时，按照手册的四层框架，从零搭建了一套完整的AI协作基础设施。这篇文章记录了我具体做了什么、产生了什么文件、以及未来30天这些基础设施将如何改变我的工作方式。

## 🔧 二、第一步：L1——建立人与AI的协作基础（实际执行：60分钟）

> 📋 **今天实际创建的文件**：`CLAUDE.md`（含7份关联规范+硬规则）+ 3条项目记忆 + 4个cron定时任务 + Git仓库初始化

### 我做了什么

**第一步：创建我的"AI使用说明书"（30分钟）**

按照手册模板，我写了一份CLAUDE.md文件，放在工作目录 `~/review/CLAUDE.md`：

```
我是谁：[你的角色和背景]
我的目标：[用AI加速什么]
关联规范文件：[7份规范文件路径，每次协作时手动加载]
硬规则：影响≥3个文件的任务，必须先输出计划并等待确认，再执行
```

这份文件的关键不是"写得多好"，而是**它让AI每次协作都有了"你的说明书"**。没有这份文件，AI只能用"平均化"的知识回应你。

**第二步：创建3条项目记忆**

在 `.claude/projects/` 下创建了3个记忆文件：
- `project-context.md`：项目用途和关联规范
- `review-standards.md`：审查质量标准
- `feedback-context-first.md`：从错误中提取的规则——"先加载上下文，否则产出质量下降约40%"

记忆文件的作用是让AI在对话结束后仍然"记得"你的规则。手册称之为"长期记忆>短期上下文"。

**第三步：设定4个定时任务——让AI在我睡觉时工作**

这是今天最大的基础设施升级。创建了4个cron任务：
- 每天 8:07 → 自动生成晨间简报（AI新闻+待办提醒+市场概览）
- 每天 22:13 → 自动Git提交（所有AI产出纳入版本控制）
- 每周六 9:17 → 五飞轮并行扫描（投资/工作/AI能力/方法论/记忆）
- 每月1日 → 月度深度回顾（五维度自评+记忆审查+规范更新）

**第四步：初始化Git版本控制**

```bash
git init && git add -A && git commit -m "init: 工作目录初始化"
```

从此，AI的每一次产出都有版本记录。任何时候我都能回答："这个文件什么时候、因为什么原因改的？"

### 第一层的成果

| 指标 | 以前 | 现在 |
|:---|:---|:---|
| CLAUDE.md（AI说明书） | 0 | 1份（含7份关联规范+硬规则） |
| 记忆文件 | 0 | 4个（含MEMORY.md索引） |
| Cron定时任务 | 0 | 4个（每日+每周+每月全覆盖） |
| Git版本控制 | 无 | 已初始化，每次产出自动commit |
| AI产出的可追溯性 | 无法追溯 | 100%（Git log记录一切） |

**💡 关键洞察**：L1的核心不是"让AI更聪明"，而是"让AI了解你的世界"。CLAUDE.md + 记忆文件 + 定时任务 = AI从"聊天机器人"变成了"工作伙伴"。

## 🚀 三、第二步：L2——让复杂任务被正确拆解和执行（实际执行：90分钟）

> 📋 **今天实际创建的文件**：Agent角色卡库（5角色+空白模板）+ 任务拆解与编排模板（含规划解耦规则）

### 我做了什么

第一层解决的是"AI能做事了"。第二层的问题是：复杂任务太大了，一个AI吃不消。而且你怎么确保AI不会"边做边偏"？

手册的原则6：**复杂任务拆解→分发→整合**。原则9：**规划与执行解耦**。

**第一步：创建Agent角色卡库**

Agent像人一样需要角色定义。我创建了5个标准化角色卡：

| 角色 | 核心职责 | 关键约束 |
|:---|:---|:---|
| 独立审查Agent | 审查产出质量，不给生成过程 | 不修改原文；不做主观判断；检查AI禁止清单 |
| 行业研究Agent | 市场/竞争/政策/技术分析 | 数据标注来源；不做投资建议；≤3000字 |
| 投资分析Agent | 多维度风险与机会评估 | 不预测股价；每个评分≤50字理由；标注置信度 |
| 内容编辑Agent | 改写为平台适配成品 | 不改变核心观点；保留修改前后对比 |
| 代码审查Agent | Python脚本审查 | 标注具体行号；P0问题无遗漏 |

每个角色卡包含：角色名称、任务、输入、产出格式、约束条件、验收标准。**约束条件比角色描述更重要**——模糊约束=产出不可控。

**第二步：创建任务拆解模板**

当一个任务涉及≥2个独立领域时，自动触发拆解：

```
任务：[一句话描述]
拆解为N个子任务：
  Agent A（角色）→ 子任务 → 产出文件 → 验收标准
  Agent B（角色）→ 子任务 → 产出文件 → 验收标准
  ...
并行启动 → 全部完成后整合 → 主报告
```

**第三步：建立"规划解耦"硬规则**

在CLAUDE.md中写入了这条规则：

> 🚫 硬规则：影响≥3个文件的任务，必须先输出计划并等待确认，再执行。

这是我今天学到的第一个教训：光靠自觉不够。必须把规则写进AI能读取的文件里——**用AI来执行AI的使用纪律。**

### 第二层的成果

| 指标 | 以前 | 现在 |
|:---|:---|:---|
| Agent角色定义 | 0（每次临时想） | 6个（5角色+1空白模板） |
| 任务拆解方式 | 全塞给一个AI | 有拆解判断流程+标准化模板 |
| 规划与执行 | 混在一起，边做边改 | 有硬规则：≥3文件必须先规划 |

**💡 关键洞察**：Agent角色卡的价值不在于"写得好"，而在于**可复用**。一次写好，每次启动Agent时复制粘贴，不再从零开始想"这次AI应该是什么角色"。

## 🔍 四、第三步：L3——建立系统化的质量保障（实际执行：90分钟）

> 📋 **今天实际创建的文件**：交叉审查SOP（4步操作流程+日志模板）+ 5领域评分矩阵库 + L1自查清单与自动修复规则

### 为什么这一层最关键

手册第一章有句话：**没有验证的AI产出是危险的。** 质量层规范更尖锐——AI产出量越大，没有系统化验证=错误率随产出量线性增长。

第二层做了多Agent并行——但并行也意味着"多个Agent同时可能出错"。没有质量网的多Agent并行=加速犯错。

### 我做了什么

**L1自查清单（所有产出必执行）**：不是"再看一遍"，而是对照清单逐条验证——

- 通用检查：文件存在？内容完整？格式正确？wikilink有效？
- 数据检查：数字前后一致？单位统一？合计=各项之和？
- 引用检查：来源标注？引用准确？日期有效？
- 合规检查：含投资建议措辞？敏感信息？政治表述？

**L2交叉审查SOP（S级/A级产出强制执行）**：这是质量层的核心——启动一个**独立Agent**审查产出，但不给它看生成过程。

操作方式：新开对话 → 只给审查Agent看最终产出+验收标准 → 让它独立输出审查报告。审查Agent不知道"生成者是怎么想的"，所以不会被生成者的逻辑带偏。

交叉审查SOP定义了完整的4步流程：准备材料→启动审查Agent→输出审查报告→根据结果行动。

**L3评分矩阵（5个领域）**：重要决策前，让独立Agent用预设矩阵打分。我创建了5个领域的评分矩阵——

| 领域 | 核心维度 |
|:---|:---|
| 投资分析 | 成长性(25%)+安全性(25%)+护城河(20%)+团队(15%)+估值(15%) |
| AI产出质量 | 完整性(25%)+准确性(25%)+清晰度(20%)+可操作性(20%)+规范性(10%) |
| 健康管理 | 科学依据(30%)+安全性(25%)+可持续性(20%)+个性化(15%)+成本(10%) |
| 事业决策 | 目标一致性(30%)+杠杆效应(25%)+能力圈匹配(20%)+时间窗口(15%)+退出成本(10%) |
| 公众号文章 | 标题(15%)+开头(20%)+信息密度(25%)+行动号召(20%)+节奏(10%)+合规(10%) |

评分Agent不知道你的偏好。**独立评分+量化矩阵=防止"我觉得这个好"的自我欺骗。**

**L4自动修复规则**：AI发现自己的错误→诊断根因→修复→验证。人不介入，除非修复失败2次。但合规问题和事实性错误不自动修——必须人确认。

### 第三层的成果

| 指标 | 以前 | 现在 |
|:---|:---|:---|
| 质量检查方式 | 凭感觉 | L1自查清单（4类17项） |
| 交叉审查 | 从未做过 | 有完整SOP（触发条件+4步流程+日志模板） |
| 评分矩阵 | 0 | 5个领域（投资/AI产出/健康/事业/公众号） |
| 自动修复 | 无 | 6类错误的修复规则+人不介入边界 |

**💡 关键洞察**：质量验证不是"不信任AI"，是"对结果负责"。飞机有双引擎不是因为不信任引擎——是因为坠机的代价太大。

## 🤖 五、第四步：L4——建立系统的自进化能力（实际执行：60分钟）

> 📋 **今天实际创建的文件**：每周反思模板（每日+每周）+ Agent监控仪表盘 + 自进化信号清单 + 首次执行反思记录

> 💡 本阶段实践的核心是L1层原则4"AI主动工作"（定时任务）+ L4层原则15-18的基础设施（反思/记忆/监控/自进化）。真正的进化能力需要数周时间积累——但第一步是先有"反思的结构"。

### 我做了什么

**反思模板（Dreaming的工程化实现）**：手册原则15——"AI需要反思时间"。我创建了两个层级的反思模板：

每日5分钟快速反思：
- 今天AI帮你做了哪些事？哪些类型出现≥2次？
- 有哪次AI产出你不满意？根因是什么？
- 有什么规则可以提取出来，让AI下次不再犯？

每周30分钟深度反思：
- 回顾本周最成功/最不满意的AI协作
- 记忆文件整理（检查腐烂、合并重复、新增gap）
- 规范文件更新（实战检验了什么？有什么缺口？）
- 五维度能力评估（有没有维度退化？）

**Agent监控仪表盘**：手册原则17——"Agent是需要管理的持续实体"。我创建了5个Agent的健康监控表，每个Agent跟踪：产出频率、质量趋势、审查发现率、指令腐化程度。

当Agent连续2周审查总是"通过"→ 审查独立性可能丧失，需要换模型或重新定义标准。
当Agent连续3次出现同类问题→ 角色卡需要更新。

**自进化信号清单**：手册原则18——终极状态是"AI自己发现问题→自己改进→自己验证"。我定义了5个到达信号：

| 信号 | 评判标准 |
|:---|:---|
| AI发现盲点 | AI主动指出你没注意到的问题 |
| AI建议改进 | AI主动说"我们可以这样改进流程" |
| 修改率<10% | 你只需审核AI产出，很少需要改 |
| 自动工作 | cron任务在你不在时正常运行 |
| 记忆自维护 | AI在反思中发现记忆冲突并建议更新 |

**首次执行反思**：在搭建完所有基础设后，我立即做了一次反思——发现了一个关键问题：**"规范先行，实战滞后"**。10个文件全部是规范和模板，但：
- ✅ 交叉审查SOP：写了——但还从未被用来审查过任何产出
- ✅ 5个Agent角色卡：定义了——但还没有一个在真实任务中使用
- ✅ 4个cron任务：创建了——但要等明天8:07才第一次触发
- ✅ 5个评分矩阵：建好了——但还没有一个被填入过实际数据

### 第四层的成果

| 指标 | 以前 | 现在 |
|:---|:---|:---|
| 反思机制 | 无 | 每日5分钟+每周30分钟模板 |
| Agent管理 | "用完即弃" | 5Agent监控表+4个调整触发条件 |
| 自进化追踪 | 无 | 5个信号清单+月度评估模板 |
| 首次反思 | 无 | 已完成——发现了"规范先行，实战滞后" |

**💡 关键洞察**：今天搭建的是"自进化的骨架"——模板、清单、监控表。但血肉需要时间来长——每周反思做30分钟，坚持4周，才能看到"系统在自我改进"的迹象。不坚持反思，L4就是空壳。

## 🔬 七、让方法论验证自身：一次真实的交叉审查

手册的质量层要求S级产出（对外发布内容）必须经过交叉审查——由一个独立Agent审查，不给它看生成过程。

既然这套方法论本身就是用来指导AI协作的，那么**用它来审查自己的产出**，就是最好的实战验证。

### 审查过程

我把今天搭建的所有10个文件、加上这篇公众号文章，提交给一个独立Agent审查。按照交叉审查SOP：新开对话 → 只给审查Agent看最终产出和验收标准 → 不给生成过程。

### 审查发现了什么

独立审查Agent发现了11个问题，按严重度排列：

| 严重度 | 数量 | 典型发现 |
|:---|:---:|:---|
| 🔴 必须修改 | 3个 | CLAUDE.md写"自动加载"但实际需手动加载；反思模板写了"6条feedback"但实际只有1条；操作日志缺少"实战验证度"声明 |

## 💡 八、五个最关键的教训

### 1. "给上下文"比"给指令"重要十倍

AI不是不聪明，是不了解你。一份CLAUDE.md（30分钟写的）比100条精美Prompt（花30天优化）的效果更好。因为前者让AI"理解你的世界"，后者只是让AI"执行你的指令"。

**一句话**：你的AI和别人用同一个AI的差别，不是你更会写指令，是你给了它别人没给的上下文。

### 2. "给目标不给步骤"是分水岭

什么时候你真正开始用好AI了？不是你学会了更多Prompt技巧的时候，是你第一次说出"完成X，标准是Y"，然后AI自己找到了比你预想的更好的执行路径的时候。

**判断标准**：你看AI的执行过程，是"嗯，跟我想的一样"还是"哦，还能这样？"——后者越多，说明你越善于用AI。

### 3. 质量验证不是"不信任AI"，是"对结果负责"

有人觉得让另一个AI审查AI产出是"不信任AI"。错了。飞机有双引擎不是因为不信任引擎，是因为坠机的代价太大。关键产出走交叉审查也是同理——不是AI不可信，是"出错的影响你承担不起"。

### 4. 定时任务是"习惯养成器"，不是"省时器"

AI的定时简报帮你省的时间其实不多（可能每天10-15分钟），但它解决了一个更大的问题：**信息到达的确定性**。你不会"忘了看"——因为你不需要记着"要看"，AI会推给你。

### 5. 基础设施 ≠ 能力——诚实面对"规范先行，实战滞后"

今天5小时搭建了10个文件+4个cron任务，看起来很完整。但独立审查一针见血：交叉审查SOP写了但没跑过一次，5个角色卡定义了但没用过一个，评分矩阵建好了但没填过一个数据。**建了≠用了。**

这是手册第十二章"常见失败模式1"的活案例。避免方法是：在新领域启用AI产出前，先定义质量检查清单——然后立刻跑一次验证。

### 6. 让方法论验证自身——用同一套标准审查自己的产出

手册的质量层要求S级产出必须交叉审查。如果我自己写方法论实践文章却不走交叉审查，那就是"只要求别人做到"。

于是我启动了独立审查Agent。它真的发现了3个我自己没注意到的问题。这证明了：**不是你不够仔细，是任何人看自己的产出都有盲点。独立视角是不可替代的。**

## ▶️ 九、如果你也想开始

你不需要5小时搭建完整体系。**今天只需要30分钟**：

### 第1步（现在，30分钟）

创建一个文件（CLAUDE.md），写下三件事：
1. 你是谁（角色、背景、你在意什么）
2. 你的工作流中，哪件事最重复、最耗时？（选1件）
3. 这件事"做完了"的标准是什么？（写3条可验证的验收条件）

### 第2步（明天，第一次尝试）

打开AI工具，先让它读你的CLAUDE.md，然后说：

> "完成 [你选的那件事]。约束：[你的限制]。验收标准：[你写的3条]。产出位置：[文件路径]。"

不要给步骤。就给它目标。

### 第3步（后天，第一次验证）

AI产出后，追加一句：

> "对照验收标准，逐条检查你的产出。标注任何不满足标准的地方。"

### 然后

如果你发现这3步让你的效率提升了，这里有两条路径：

- **快速路径**：按照本文第二至五节的步骤，花5小时搭建完整的四层基础设施。我已经把每一步的具体命令和文件模板整理成了《[AI方法论从0到1可复现操作日志](操作日志/AI方法论从0到1_可复现操作日志.md)》，你可以直接复制粘贴。
- **进阶路径**：读完整的《AI方法论手册》v1.2（18条原则+四层框架），理解"为什么这样做"之后再动手。
