> 核心理念：编程已被解决——但你管理和指挥AI的方式，决定你的上限

## 零、Cherny是谁

Boris Cherny，Anthropic公司Claude Code创始人兼负责人。前Meta首席工程师（7年），著有《Programming TypeScript》一书。

**2026年的Cherny**：
- 自2025年10月起，**零行手写代码**
- Claude Code代码库100%由AI自动生成
- 每天通过手机发起数十个PR，单日最高纪录**150个PR**
- 夜间运行**数千个Agent**并行工作
- 管理5-10个活跃编码会话，每个下挂多个Agent

用他自己的话说：**"编程已被解决。"** 不是夸张——是对他所在技术栈（TypeScript + React）的事实陈述。

## 一、Cherny的日常工作流

### 1.1 一天的全景

```
早晨
  └─ 打开Claude手机应用
  └─ 左侧Code标签，5-10个活跃会话
  └─ 检查昨夜Loop运行的Agent产出
  └─ 审核PR、合并、分配新任务

白天
  └─ 碎片时间（会议间隙/通勤/排队）发指令
  └─ 每个指令→启动一个Agent执行
  └─ 系统通知：Agent完成→切过去审核→下一个指令
  └─ 像玩星际争霸一样管理Agent群

夜间
  └─ 数千个Agent并行启动深度任务
  └─ Loop/Routines持续运行（CI修复、数据采集、测试维护）
  └─ Agent间通过Slack互相通信协作
```

### 1.2 核心工作模式：不是写代码，是"审核PR"

Cherny的工作已经从"编写代码"完全转变为**审核和合并AI生成的PR**。他的角色变化：

| 以前（2025年前） | 现在（2026年） |
|:---|:---|
| 写代码 | 定义任务方向 |
| 调试 | 审核PR差异 |
| 研究技术方案 | 判断AI的方案是否合理 |
| 手写测试 | 定义验收标准 |
| 开会讨论架构 | 让Agent讨论→人做最终决策 |

### 1.3 手机 = 主力工作设备

Cherny的工作完全围绕手机展开。他通过Claude iOS应用：

- 在任何碎片时间发起PR指令
- 一个会话 = 一个开发任务，每个会话下挂多个Agent并行
- 系统通知告知Agent完成→审核→合并或打回
- 不需要坐在电脑前——**设备从"生产力工具"变成了"通知终端"**

## 二、三大核心机制

### 2.1 Loop：基于Cron的Agent调度系统

Loop是Cherny工作流的发动机。它基于标准cron表达式，让Agent按时间表自动执行任务。

**Cherny部署的Loop实例（部分已知）**：

| Loop任务 | 频率 | 作用 |
|:---|:---|:---|
| CI看护 | 持续/高频 | 监控PR状态，CI失败自动修复并rebase |
| 偶发测试修复 | 持续 | 发现flaky test自动诊断→修复→验证 |
| 用户反馈聚类 | 每30分钟 | 抓取Twitter用户反馈，自动分类整理 |
| 代码审查 | PR创建时触发 | 自动审查代码质量、安全、风格 |
| 依赖更新 | 每日 | 检查依赖版本、自动升级+测试 |
| 夜间深度任务 | 夜间 | 数千Agent并行执行重构、优化、文档生成 |

**Cherny的Loop哲学**：
> "不要等出问题再修。让Agent持续看护，发现问题→修复→验证，人只看异常报告。"

### 2.2 Routines：服务端常驻Agent

Loop解决"定时触发"，Routines解决"持续在线"。

Routines是运行在服务器上的常驻Agent实例。Cherny合上笔记本电脑后，Routines继续工作。第二天早上打开手机，看到的是一夜之间Agent产出的PR列表和异常报告。

**Routines与Loop的区别**：

| 机制 | 触发方式 | 适合场景 |
|:---|:---|:---|
| Loop | 定时（cron） | 周期性任务：监控、采集、日报 |
| Routines | 事件/持续运行 | 实时响应：CI监听、告警处理、数据管道 |

### 2.3 Multiagent并行

Cherny同时运行5-10个终端tab，每个tab是一个独立的Claude Code实例。加上夜间数千个Agent，他的管理方式被形容为**"更像玩星际争霸而不是编程"**——注意力在Agent群之间切换，每个Agent完成任务后通过系统通知"呼唤"他审核。

**Agent间通信**：不同Claude实例通过Slack互相通信。一个Agent完成任务后通知另一个Agent，形成协作链。

## 三、CLAUDE.md：让代码库自我修正

### 3.1 这是Cherny最重要的基础设施

CLAUDE.md是存放在每个代码仓库根目录的共享指令文件。Cherny将其视为**代码库的"操作手册"**——每次Agent犯错，就把修正规则写入CLAUDE.md，从此所有Agent自动遵守。

### 3.2 CLAUDE.md的内容结构（Cherny风格）

```
# 项目概述
[项目是什么、核心目标、技术栈]

# 编码规范
- 命名约定
- 文件组织方式
- 测试要求
- 安全红线

# 常见错误（最重要的部分）
- 错误1：描述 + 为什么会犯 + 正确做法
- 错误2：...
- [每次Agent犯错后追加]

# 验收标准
- PR合并条件
- 测试覆盖率要求
- 性能基线
```

### 3.3 Cherny的CLAUDE.md管理原则

1. **即时更新**：Agent犯错→立刻追加到CLAUDE.md→所有Agent下次自动避免
2. **具体而非抽象**：不写"代码要好"，写"不用`any`类型、函数不超过50行、每个PR附带测试"
3. **积累是复利**：CLAUDE.md越厚，Agent犯错越少。前6个月积累的规则=永久可复用的智力资本
4. **共享上下文**：所有Agent启动时先读CLAUDE.md→不用每次重新解释项目规则

**Cherny的原话**：
> "把规则写进CLAUDE.md，让代码库成为一个自我修正的有机体。"

## 四、Cherny的两条黄金法则

### 法则1：给目标，不给步骤

```
❌ "先打开src/utils.ts，找到formatDate函数，改成ISO格式，然后更新所有调用处"
✅ "把项目中所有日期格式统一为ISO 8601，更新所有调用处，跑测试确认"
```

原因：AI自己规划的路径往往比人指定的更优。人定义"要什么"，AI定义"怎么做到"。

### 法则2：信任执行，验证结果

Cherny不盯着Agent的执行过程。他只看最终PR的diff。省下的注意力用于判断：这个改动方向对吗？边界情况处理了吗？测试够吗？

**关键**：不盯过程≠放任不管。Cherny对PR的审查非常严格——他只是不浪费时间看Agent的"中间步骤"。

## 五、Cherny的Agent管理哲学

### 5.1 Agent不是工具，是需要管理的持续实体

Cherny不把Agent当成"用完即弃"的一次性执行器。他把Agent当成需要持续监控、评估、进化的实体：

- **监控**：每天早上检查Agent群的产出报告
- **评估**：这个Agent的PR质量在上升还是下降？
- **进化**：发现Agent反复犯同类错误→更新CLAUDE.md→所有Agent自动升级

### 5.2 失败自动修复，不把错误丢回给人

Cherny的Loop中有专门"CI看护"实例——PR合并后CI失败，Agent自动：
1. 读取CI失败日志
2. 诊断根因（不是症状）
3. 修复代码
4. 运行测试验证
5. Push + 记录修复过程

人只在修复失败2次后才介入。

### 5.3 并行是默认，串行是例外

Cherny的原则：能并行的绝不串行。5个独立子任务=5个并行Agent。合并冲突？让另一个Agent去解。

**管理并行的技巧**：
- 每个Agent任务独立、边界清晰（不共享中间状态）
- 任务粒度为"30分钟到2小时"可完成
- 超过2小时的任务进一步拆解

## 六、Cherny的核心产品哲学

### 6.1 "为6个月后的模型构建，不为今天的模型构建"

这是Cherny最重要的产品设计原则。Claude Code当前需要的一些能力（如更复杂的推理链），如果判断6个月后的模型会天然具备，就不在今天的产品中投入工程资源去弥补。

**对你的启示**：选AI工具时，不仅要看它今天能做什么，更要判断它的架构是否为即将到来的更强模型预留了空间。

### 6.2 "Claude Code最终可能只剩100行代码"

Cherny认为，随着模型能力增强，Claude Code这个"外壳"会越来越薄。模型越强，需要的工程胶水越少。最终Claude Code可能只是一个极薄的、连接模型与操作系统的接口。

### 6.3 "软件工程师头衔将消失，被Builder取代"

Cherny预测：未来的Builder是跨学科通才——同时懂产品、设计、数据、工程。AI压缩了每个领域的入门成本，一个人可以覆盖原来一个团队的技能面。

## 九、常见问题

**Q: Cherny真的不写代码了吗？**
A: 是的。自2025年10月起，Claude Code代码库100%由AI生成。Cherny的工作是审核PR、定义任务、管理Agent群、更新CLAUDE.md。

**Q: 我需要成为顶级工程师才能这样工作吗？**
A: 不需要。Cherny的工作方式不需要你比他更懂AI——需要你建立CLAUDE.md、学会给目标不给步骤、信任Agent执行、用Loop自动化重复任务。这些都是行为转变，不是技术门槛。

**Q: 每天150个PR，质量怎么保证？**
A: 三个机制——CLAUDE.md（规则前置）、Loop自动修复（失败自愈）、Cherny的最终审核（每个PR他都看diff）。不是"AI随便写→直接合并"。

**Q: 手机真的能替代电脑开发吗？**
A: Cherny用手机完成的是"任务指派+PR审核"——不是用手机写代码。他的转变核心是：从"亲自编写"变为"指挥+审核"。
