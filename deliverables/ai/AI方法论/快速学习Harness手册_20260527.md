## 零、一个前提：Harness不只是工具，是软件交付的"自动驾驶"

大多数人理解CI/CD就是"代码自动上线"。Harness做的是**从代码提交到生产运行的全生命周期自动化**——不只是部署，还包括测试、安全、成本、可靠性。

**Harness = AI驱动的软件交付自动化平台。** 记住一句话：软件领域的"自动驾驶"。

| 角色 | 以前 | 有了Harness |
|------|------|-----------|
| 开发者 | 写代码→等测试→等运维→等安全，上线2周 | 代码提交→全自动，当天上线 |
| 测试团队 | 手跑用例，80%时间测无关功能 | AI自动识别影响范围，只测该测的 |
| 运维 | 凌晨部署，出错手工回滚1小时 | 金丝雀自动发布，秒级自动回滚 |
| 安全团队 | 上线后扫描，漏洞已暴露 | 代码提交时自动扫描，提交即拦截 |
| 管理者 | 下月账单才知道云花了多少 | 实时看到每个团队、每个服务的云成本 |

**检查点**：你能不能用一句话说出Harness解决的核心问题？

## 一、第1层：建立正确认知——Harness改变了谁的工作

> 这一层解决的是：Harness是什么、为什么现在重要、和已有的知识怎么关联。

### 1.1 场景驱动：银行APP更新一个功能

| 步骤 | 没有Harness | 有Harness |
|------|-----------|----------|
| 测试 | 测试团队手跑3天，测1000个用例，只有50个相关 | AI自动识别50个相关用例，**10分钟跑完** |
| 部署测试环境 | 运维手动操作半天，可能搞错环境配置 | **一键自动部署** |
| 部署生产环境 | 凌晨加班，出错回滚1小时 | 金丝雀发布：5%→50%→100%，**秒级回滚** |
| 安全检查 | 上线后发现漏洞，已经暴露 | **提交时自动扫描**，有问题立刻拦截 |
| 云费用 | 下月账单来了，不知道谁花的 | **实时显示**每个团队、每个服务的成本 |

### 1.2 Harness vs Jenkins：为什么是现在

Jenkins诞生于2011年，Harness是AI原生新一代。

| 维度 | Jenkins | Harness |
|------|---------|---------|
| AI能力 | 没有 | **AI原生**：自动识别测试范围、自动回滚、AI生成流水线 |
| 维护成本 | 需要专人维护插件和脚本 | 大部分开箱即用 |
| 部署策略 | 手写脚本 | 内置蓝绿/金丝雀/滚动6种策略 |
| 云成本可视 | 无 | **内置**，客户平均降低60%云成本 |
| 适用模式 | 有专职DevOps团队 | 开发团队自服务 + 平台团队治理 |

> 投资视角：看一家科技公司用Jenkins还是Harness，是判断其工程成熟度的信号。

### 1.3 和你已有知识的关联

| 你已知的 | 对应Harness中的 | 关联 |
|---------|--------------|------|
| Claude Code | Harness AI模块 | 两者都是AI驱动工作流。Harness 2026年AI模块全面GA |
| Git | CI模块 | 代码push到Git → Harness自动触发构建→测试→部署 |
| Docker/K8s | CD模块 | Harness部署的目标环境通常是K8s集群 |
| 科技金融尽调 | 技术成熟度评估 | 看企业CI/CD工具链可判断其工程化水平 |

## 二、第2层：组织核心概念——6个模块+3个核心机制

> 这一层解决的是：Harness由哪些部分组成，各自干什么。不要一次学完，先记6个缩写。

### 2.1 六大模块速记

| 缩写 | 全称 | 一句话 |
|------|------|--------|
| **CI** | Continuous Integration | 代码一提交，自动构建+测试 |
| **CD** | Continuous Delivery | 测试过了，自动部署上线 |
| **FF** | Feature Flags | 新功能先给1%用户试，没问题全开 |
| **STO** | Security Testing | 代码里有漏洞？提交时就告诉你 |
| **CCM** | Cloud Cost Mgmt | 谁多花了云的钱，一目了然 |
| **SRM** | Service Reliability | 出问题AI先分析，不用人半夜爬起来 |

### 2.2 三个核心机制

**Pipeline（流水线）= 自动化流程**

```
代码提交 → [CI: 构建 → 测试 → 安全扫描] → [CD: 部署测试 → 部署生产]
```

类比银行审批：客户提交→客户经理初审→风控复核→分行审批→放款。每步有成功/失败判断。

**Delegate（代理）= 执行任务的工人**

Harness控制面在云端（SaaS），但执行任务的Delegate部署在你自己的服务器上。Delegate只**主动连接**控制面，不需要开放防火墙入站端口——这是金融行业最关心的安全模型。

**Template（模板）= 可复用的流程标准**

平台团队定义"标准上线模板"：必须经过哪几步、每步什么检查。应用团队在模板上填写参数，不能绕过安全检查——这就是**治理和控制**。

**检查点**：不看上文，说出6个模块缩写和3个核心机制。

## 三、第3层：质量保障——Harness的安全与治理模型

> 这一层解决的是：用Harness不只是快，还要安全、可控、可追溯。

### 3.1 安全左移（Shift Left）

传统安全是"上线前扫一遍"——太晚了。Harness在**代码提交时**就自动扫描：

```
代码提交 → [STO自动扫描] → 有漏洞？→ 直接拦截，不给过
                              → 没漏洞？→ 继续走CI/CD流程
```

### 3.2 部署安全：金丝雀发布

不是一次性全量上线，而是渐进式：

```
5%用户 → 监控5分钟无异常 → 50%用户 → 监控5分钟无异常 → 100%全量
                                                          ↓
                                                    任何一步出问题
                                                          ↓
                                                    秒级自动回滚
```

### 3.3 治理控制：模板+审批

- 平台团队定义标准模板，应用团队不能绕过
- 关键步骤可设置人工审批节点
- 所有操作可审计、可追溯

### 3.4 金融行业案例

| 机构 | 做什么 | 效果 |
|------|--------|------|
| **花旗银行** | 全球应用部署平台 | 部署数天→7分钟，构建加速8倍，云成本降35% |
| **某美国大行** | 合规+安全+CI/CD全流程 | 满足SOX/PCI-DSS合规要求 |
| **多家金融科技** | 微服务持续交付 | 发布频率从月级→日级 |

> 实用意义：如果向银行推荐DevOps方案，花旗案例是最强的背书。

**检查点**：你能说出Harness保障质量的三个机制吗？（安全左移、金丝雀发布、模板治理）

## 四、第4层：持续进化——评估、优化、扩展

> 这一层解决的是：怎么判断用得好不好，怎么持续优化。

### 4.1 四个评估维度（DORA指标）

| 指标 | 精英团队水平 | 你要问的问题 |
|------|------------|------------|
| 部署频率 | 每天多次 | "你们多久部署一次？" |
| 提交到上线时间 | <1小时 | "从代码提交到上线要多久？" |
| 变更失败率 | 0-15% | "有没有自动回滚？回滚要多久？" |
| 故障恢复时间 | <1小时 | "出问题到恢复要多久？" |

### 4.2 评估一家公司用Harness的水平

| 问题 | 你能得到什么信息 |
|------|----------------|
| "你们用Harness的哪些模块？" | 深度——只用了CI还是全模块 |
| "部署频率多少？提交到上线平均时间？" | DORA成熟度 |
| "有没有自动回滚？回滚需要多久？" | Harness核心优势是否用到位 |
| "云成本管理用的是什么？" | 成本治理意识 |
| "安全扫描是上线前还是上线后？" | 安全实践成熟度（好的是提交时即扫） |

### 4.3 免费起步

```bash
# 一台有Docker的电脑，一条命令
docker run -d -p 3000:3000 \
  -v /var/run/docker.sock:/var/run/docker.sock \
  --name harness harness/harness
```

打开 `http://localhost:3000`，账号 `admin`，密码 `changeit`。5分钟看到界面。免费计划含每月2000次构建额度。
