# AI安全层规范

> 基于Microsoft Agent Control Specification (ACS) 的个人AI方法论安全层
>
> 版本 v1.0 | 2026年6月3日 | 跨层约束规范，贯穿L1-L4四层框架

---

## 一、为什么需要安全层

### 1.1 问题：AI Agent的能力增长快于行为约束

2026年6月，微软发布**Agent Control Specification (ACS)**——全球首个开源AI Agent行为约束标准。这不是巧合：当AI Agent从"单次问答"进化为"多步骤自主执行"，安全问题从理论风险变成日常挑战。

你的Claude Code日常操作权限包括：**读写文件、执行Shell命令、管理Git仓库、调用外部API**。每项能力本身都有用，但组合起来——一个误指令就可能触发一系列不可逆操作。

### 1.2 安全层的定位：贯穿四层的"约束线"

安全层**不是一个独立层级**（不打破L1-L4结构），而是一条贯穿所有层的约束规范：

```
L1 关系层 ──┐     谁有权做什么（关系定义包含权限）
L2 指令层 ──┤     指令中包含约束条件（什么不能做）
L3 质量层 ──┤ ←── 审计追溯挂在质量验证链上
L4 进化层 ──┘     安全事件作为进化层的反馈输入
```

### 1.3 目标

- **可操作**：每一条规定都可以在今天、在Claude Code中落地
- **可衡量**：是否有安全违规可以明确判断
- **可进化**：安全规则随使用场景和AI能力变化而调整

## 二、核心框架：个人版三层模型

微软ACS（Agent Control Specification）定义了基于5个检查点的技术框架——在Agent的输入、大模型调用、状态改变、工具执行、输出环节设置YAML声明式安全策略。受ACS的"分层检查点"理念启发，我将个人AI Agent的行为规范抽象为一个更适合个人用户的三层模型：

| 个人三层模型 | 方法论命名 | 核心问题 | 对应权力 |
|---------|-----------|---------|---------|
| 权限分级 | **能力边界** | Agent能做什么、不能做什么 | 立法权（人定规则） |
| 审计追溯 | **行为记录** | Agent做了什么、怎么做的 | 司法权（系统记录） |
| 人工干预 | **决策门** | 什么需要人确认、什么自动停 | 行政权（人执行监督） |

### 2.1 能力边界（权限分级）

将Agent操作按风险从低到高分为四级：

| 级别 | 名称 | 定义 | 默认策略 | 风险示例 |
|:----:|:----|:----|:--------|:--------|
| **L0** | Read | 读取文件、搜索内容、查看状态 | ✅ 自动执行 | 读取投资记录（信息泄露风险，但无破坏性） |
| **L1** | Write+Notify | 创建文件、修改已有文件、提交Git | ✅ 自动执行 + 通知用户 | 修改方法论文档（需要通知但不必阻塞） |
| **L2** | Execute | 运行Shell命令、调用API、执行脚本 | ⛔ 每次请求确认 | rm、git push、pip install |
| **L3** | Delete | 删除文件、覆盖重要文档、不可逆操作 | ⛔ 双重确认 + 备份检查 | rm -rf、覆盖关键文档 |

**映射到Claude Code的权限模式**：
- **Plan模式** → L0 + L1（读+写，但标记为计划，不执行）
- **Auto模式** → L0 + L1（自动执行低风险操作）
- 高风险命令（rm、git push等）→ 始终触发L2确认，即使Auto模式

**个人行为规范**：
1. 新任务开始时，明确该任务的最高权限级别
2. 权限升级（从L1到L2）必须有人为确认
3. 涉及金融、法律、对外发布的操作，自动提升一级权限要求

### 2.2 行为记录（审计追溯）

每个Agent操作留下可追溯的记录链：

```
Agent操作 → 时间戳 → 操作类型 → 影响文件 → Git Hash → 审查结果
```

**记录内容**：
- 每次文件写入：时间 + 文件路径 + 操作类型（新建/修改/删除）
- 每次Git操作：Commit message + 变更摘要
- 每次Shell执行：命令 + 退出码 + 输出摘要
- 每次质量审查：审查Agent ID + 审查结果 + 发现的问题

**追溯能力要求**：
- 任何文件变更可以追溯到"谁在什么时间因什么任务修改了什么"
- S级/A级产出的审查链完整可查
- 投资决策相关操作需额外记录"决策理由"字段

**Git作为追溯基础设施**：
- 每次AI写入后自动commit（原则14：版本控制一切）
- Commit message标注产出Agent和任务描述
- 关键决策节点打tag

### 2.3 决策门（人工干预）

在关键节点设置不可绕过的"确认门"：

| 场景 | 干预类型 | 触发条件 |
|:----|:--------|:--------|
| 文件删除 | ⛔ 双重确认 | 任何`rm`/`Delete`操作 |
| 投资决策 | ⛔ 双重确认 + 独立评分 | 买卖/加减仓指令 |
| 对外发布 | ⛔ 双重确认 | 公众号文章群发前 |
| 大规模修改 | ⚠️ 计划确认 | 影响≥3个文件的任务 |
| 首次操作新领域 | ⚠️ 一次性确认 | 首次在非投资领域执行写操作 |
| 异常模式 | ⛔ 自动暂停 | 检测到重复失败/资源消耗激增 |

**决策门实现方式**：
- Claude Code的权限弹窗（已有）：L2-L3级别操作自然触发
- 任务启动前声明权限级别：在指令中明确"本任务最高权限为L1"
- 操作后确认回执：S级产出完成后要求用户确认"接受所有变更"

## 三、与四层框架的关系

### 3.1 与L1（人与AI的关系）的关系

L1定义"你和AI谁做什么"。安全层的**能力边界**规则给出了具体的权限粒度：

- L1原则1（给AI权限）→ 安全层细化：给到什么级别的权限？Read还是Execute？
- L1原则5（信任但要验证）→ 安全层提供验证基础：审计记录作为信任的凭证

**实战示例**：
> "AI有权访问我的投资文档"（L1）→ "但删除文件前必须问我"（安全层L3权限）

### 3.2 与L2（指令层）的关系

每条指令中嵌入"约束条件"，而非单独的安全规则文件：

- 标准指令模板新增"约束条件"字段（+ "禁止操作"字段）
- 复杂任务的指令中包含明确的权限边界声明

**实战示例**：
> "分析昆仑万维Q1财报。你只能读取投资目录下的文件（L0），不能执行脚本（L1+）。分析结果以Markdown写入deliverables/目录（L1），不覆盖已有文件。"

### 3.3 与L3（质量层）的关系

审计追溯嵌入质量验证链：

- 每一次质量审查（L2交叉审查、L3独立评分）自动记录审计行
- 质量层发现的"逻辑矛盾"可能触发安全层的"人工干预"
- S级产出：质量层五层全跑 + 安全层审计追溯全开

### 3.4 与L4（进化层）的关系

安全事件作为进化反馈：

- 每一次安全违规/差点违规 → 记录到经验教训库
- 安全规则随使用场景自动调整（原则18：自进化）
- 安全层自身也需要定期审查：是否有过期规则？新增场景是否需要新规则？

## 四、Claude Code安全配置实操

### 4.1 利用Claude Code现有权限系统

Claude Code已有内置权限模型，安全层规范在此基础上做系统化映射：

| Claude Code内置机制 | 安全层映射 | 操作建议 |
|-------------------|-----------|---------|
| Tool use权限弹窗 | L2 Execute确认 | 不要无脑点"Allow"，确认命令内容后再允许 |
| Auto模式 | L0+L1自动 | 在Auto模式下检查产出时不跳过终审 |
| Plan模式 | 全部L0 | 复杂任务先用Plan模式出计划，审阅后再切换 |
| Git自动commit | 审计追溯 | 接受commit前检查变更摘要 |

### 4.2 权限分级清单（可打印）

```
┌─────────────────────────────────────────────┐
│           AI Agent 权限速查表                 │
├──────────┬──────────┬───────────┬────────────┤
│  操作     │  风险     │  默认策略  │  确认方式   │
├──────────┼──────────┼───────────┼────────────┤
│  读文件   │  L0 低    │  ✅ 自动   │  无需确认    │
│  搜索     │  L0 低    │  ✅ 自动   │  无需确认    │
│  写文件   │  L1 中    │  ✅ 自动   │  ✓ 检查    │
│  修改文件  │  L1 中    │  ✅ 自动   │  ✓ 检查    │
│  Shell命令 │  L2 高    │  ⛔ 确认   │  ☝ 每次确认  │
│  Git操作   │  L2 高    │  ⛔ 确认   │  ☝ 每次确认  │
│  删除文件  │  L3 极高   │  ⛔ 双重确认 │ ✋ 双重确认  │
│  不可逆操作 │  L3 极高   │  ⛔ 双重确认 │ ✋ 双重确认  │
└──────────┴──────────┴───────────┴────────────┘
```

### 4.3 Hook脚本示例

利用Claude Code的Hook机制实现自动约束：

```bash
# pre-commit hook: 检查是否包含敏感信息
if grep -qi "密码\|password\|token\|secret\|api_key" "$@"; then
  echo "⚠️ 检测到可能的敏感信息，请确认后提交"
  exit 1
fi
```

## 五、个人Agent行为规范（操作版）

### 5.1 三条基本原则

1. **清楚告诉AI它的权限边界** — 任何新任务启动时，在指令中明确权限等级
2. **高风险操作永不自动** — 涉及删除、资金、对外说的操作，必须人工确认
3. **所有操作可追溯** — 每项变更都能回答"谁在什么时候改了什么东西"

### 5.2 每日自检三问

```
□ 今天AI做了什么我没有检查的操作？
□ 今天有没有"差点出事"的时刻？
□ 今天的哪些操作需要加强约束？
```

### 5.3 安全事件记录模板

```
事件：删除文件差点误操作 / Agent越权访问 / 其他
日期：YYYY-MM-DD
场景：描述上下文
严重程度：1-5（5最严重）
原因：为什么发生了
措施：如何防止再次发生
```

## 六、安全自检清单

### 任务启动时

- [ ] 明确本任务最高权限级别（L0-L3）
- [ ] 关键限制已嵌入指令（"不能删除已有文件"等）
- [ ] 涉及投资/对外发布/个人信息的操作已标记

### 任务执行中

- [ ] L2操作每次都确认再放行
- [ ] L3操作双重确认后再执行
- [ ] 发现异常立即暂停（Ctrl+C / "stop"）

### 任务完成后

- [ ] 检查Git变更摘要，确认没有意外修改
- [ ] 确认没有创建/删除意料之外的文件
- [ ] 一次安全回顾：今天有"差点出事"吗？

### 周度

- [ ] 审查本周安全事件记录（如有）
- [ ] 检查权限规则是否需要更新
- [ ] 检查AI使用的服务/API密钥是否安全
