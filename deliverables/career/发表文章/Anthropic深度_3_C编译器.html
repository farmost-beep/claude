大多数人对AI编程的想象是：一个人坐在电脑前，用自然语言描述需求，AI帮你写代码。


Anthropic的研究员Nicholas Carlini做了一个截然不同的实验。他让16个Claude Agent同时工作——全都访问同一个代码仓库，通过一个简单的文件锁定机制同步——目标是：从零开始构建一个能编译Linux内核的C编译器。


结果出人意料。


实验设计


16个Claude Agent被丢进一个无限循环中：


```bash

while true; do

claude --context-file context.md --execute prompt.md

done

```


每个Agent运行时，读取当前任务描述，在共享代码库中工作，然后通过一个文件锁定机制（类似数据库中的锁）来避免冲突。


所有Agent共享：

- 同一个Git仓库

- 一个`context.md`文件（记录当前状态和目标）

- 一个任务队列（每个Agent完成一个任务后领取下一个）


这不是一个精心编排的分布式系统——这是一个原始的、各凭本事的协作模式。


结果


经过约2000次Claude Code会话，花费约2万美元的API调用成本，16个Agent共同生成了一个10万行Rust代码的C编译器。


这个编译器成功编译了Linux 6.9内核，支持x86、ARM和RISC-V三种架构。


作为对比：一个生产级C编译器通常由一个数十人的工程团队花费数年时间开发。16个Agent在数周内完成了一个可工作的版本。


三条经验


Carlini在文章中总结了三条核心经验：


1. 写极其高质量的测试。


这是最重要的发现。当16个独立的Agent在同一个代码库上工作时，它们之间没有实时沟通。每个Agent都不知道其他Agent正在做什么。唯一能防止"一个人改坏了另一个人的代码"的机制就是测试。


Carlini的观察：测试不仅仅是验证代码是否正确——测试是多Agent系统中的通信协议。一个Agent写的测试告诉其他Agent："这段代码的预期行为是这个。"


2. 针对AI的局限性设计，而非对抗它。


Carlini发现了Claude在长时间运行中的几个弱点：


- 上下文污染——随着对话变长，Agent会忘记自己已经做了什么

- 时间盲区——Agent无法准确估计一个任务需要多长时间

- 过度自信——Agent倾向于认为自己的代码是正确的


设计策略不是"让AI克服这些弱点"，而是"让系统不依赖于AI在这些维度上的表现"。例如：每个Agent的任务都限制在2-4小时内完成，避免长时间上下文污染。


3. 用GCC作为正确性预言机。


编译器开发的独特优势是：有一个现成的正确性标准——GCC。Agent编写代码后，可以用GCC的输出作为标准答案来验证自己的编译结果是否正确。不是所有项目都有这样的条件。


超出预期的意义


这个实验的意义不在于"16个Agent做了一个编译器"。更有趣的是它对多Agent协作的洞见：


- Agent不需要复杂的协调协议。一个共享文件加一个锁定机制就够了。

- 测试是Agent之间最重要的沟通方式。

- 多Agent系统的瓶颈不是技术，是如何防止Agent互相干扰。


Carlini的实验方法也很说明问题：他不是先设计一个精密的架构，而是用一个原始的原型跑起来，然后观察问题、调整、再跑。这种"先跑起来再看"的方法本身，就是AI Native工程的典型方式。
