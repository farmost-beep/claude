# AI能力发现方法论

> 陈颖芳 | 2026年5月
>
> 本文约4000字，阅读时间约11分钟。本文仅代表作者个人观点。
>
> 方法论系列第十三篇。前十二篇建立了从认知到落地的完整体系，并横向评价了主流产品。这一篇回答一个更底层的问题——在这个AI能力以周为单位迭代的时代，你怎么系统化地知道"AI现在能做什么"？

---

## 这个问题的紧迫性

一个事实：**大多数人对自己使用的AI产品的真实能力边界，了解程度不超过30%。**

不是因为他们不够聪明。是因为三个结构性问题：

1. **信息洪流**：每天几十篇论文、十几个新产品、上百条"AI又突破了"的推文——你不可能全看
2. **炒作噪音**：每个发布都号称"颠覆"，每个demo都看起来完美——你不知道哪个是真的
3. **能力颗粒度不匹配**：你知道"AI会写代码"，但不知道它具体在哪种代码、哪个复杂度、哪个环节最擅长——而正是这种颗粒度的了解，决定了你用得好不好

**AI能力发现的本质不是"多知道一些"，是"知道得比噪音更精确"。**

这篇文章给一套系统化的方法论。

---

## 第一层：建立四源信息体系

信息源不是越多越好。是每个源回答一个特定的问题。四个源就够了：

### 源一：基准和榜单（回答"AI现在客观上能做到什么程度"）

不要看产品宣传页。看基准测试排名和第三方评测。

| 信息类型 | 具体来源 | 更新频率 | 回答的问题 |
|:---|:---|:---:|:---|
| 编码能力 | SWE-bench Verified 排行榜 | 月度 | AI能独立修多难的bug？ |
| 推理能力 | GPQA Diamond、MMLU-Pro | 季度 | AI的推理到了什么水平？ |
| 长上下文 | RULER、Needle-in-a-Haystack | 新模型发布时 | 长文档理解真的可靠吗？ |
| Agent能力 | WebArena、OSWorld | 季度 | AI能自主操作软件吗？ |
| 多语言 | MGSM、FLORES | 新模型发布时 | 中文/日文能力如何？ |

**操作**：每季度花30分钟扫一遍主要基准排名。不需要看每篇论文，只需要知道"目前第一名是谁、比三个月前提升了多少、在哪个维度提升最大"。

最关键的一条规则：**看基准不要只看总分。** 总分掩盖了能力的长短腿。要看子项得分——一个模型总分提升了10%，但如果代码能力下降了而推理能力飙升了，对你用不用得上的判断完全不同。

### 源二：产品变更日志（回答"我能用上什么新东西"）

比论文更重要的信号源。因为论文到产品有6-18个月的滞后，而产品变更日志是你立刻能用上的。

| 信息类型 | 具体来源 | 频率 | 重点看什么 |
|:---|:---|:---:|:---|
| Claude Code 更新 | Anthropic Release Notes | 双周 | 新工具、新Agent能力、新记忆功能 |
| Cursor / Copilot 更新 | 产品Changelog | 月度 | Agent模式演进、新集成 |
| 模型能力更新 | 各厂商Blog | 新模型发布 | 能力对比表、实际benchmark |
| MCP / 插件生态 | GitHub Trending | 持续 | 新工具接入、新使用模式 |

**操作**：订阅三个RSS/Newsletter：Anthropic Blog、OpenAI Blog、GitHub Trending（AI相关）。每周10分钟扫标题，看到"新功能"才点进去。

### 源三：实战案例（回答"别人是怎么用的，效果真的好吗"）

基准测试告诉你AI能做什么，实战案例告诉你AI在真实场景中能做什么——两者之间通常有30-50%的衰减。

| 信息类型 | 最佳来源 | 为什么这个源好 |
|:---|:---|:---|
| 工程实践 | Hacker News 的AI相关讨论 | 评论区的质疑比文章本身更有价值 |
| 投资研究 | Substack上的独立分析师 | 独立作者比机构报告敢说真话 |
| 产品设计 | X(Twitter)上的一线从业者 | 真实工作流 > 官方Demo |
| 学术突破 | arXiv Sanity Preserver | 按热度排序，天然过滤噪音 |

**操作**：关注10-20个"一手实践者"而不是"AI评论家"。判断标准——这个人的Twitter/GitHub里有没有他自己动手跑过的代码和真实结果，还是只有转发和评论。

### 源四：反向信号（回答"什么被过度炒作了"）

最被低估的信息源。知道什么不行，比知道什么行，更有决策价值。

反向信号的最佳来源：
- **产品被关闭/停更的公告**（比正面信号诚实100倍）
- **大厂内部的AI使用率数据**（如果有渠道的话——大多数内部AI工具的使用率远低于外部宣传）
- **失败案例的复盘**（"我们用AI做X，试了三个月，失败了，原因是Y"——这种内容比"我们用AI提升了10倍效率"有价值得多）

**操作**：每当你看到一个"AI改变了X"的热门文章，搜索"AI failed X"或"X AI limitations"。正反两面的信息同时看，比只看一面更有判断力。

## 第二层：能力评估四步法

发现了一个新能力/新工具后，怎么判断它值不值得你花时间？四步：

### 第1步：能力定位——它在"能力成熟度曲线"的哪个位置？

每项AI能力都经过四个阶段。你在哪个阶段介入，决定了你投入多少时间：

```
阶段一：论文突破（学术圈）
  → 看到论文标题很兴奋 → 实际：90%不会变成可用产品
  → 你的策略：记录+等待。不投入时间。

阶段二：产品原型（早期采用者）
  → 有demo了，但只对特定场景效果惊艳
  → 你的策略：花15分钟跑一次demo。判断"在我的场景中潜在价值多大"

阶段三：产品化（可稳定使用）
  → 有API/产品了，文档完善了，有bug但不多
  → 你的策略：花1-2小时深度测试你的真实场景。如果不通过，等下一版

阶段四：基础设施（默认可用）
  → 已经成为标配，所有人都在用
  → 你的策略：不要再花时间"评估"了。直接用。你不会花时间"评估搜索引擎有没有用"
```

**核心规则**：永远不要在阶段一投入超过记录的时间。永远不要在阶段四还在"观望"。

### 第2步：能力边界测试——用你的真实场景，不是官方的demo

官方demo是最差的能力评估工具。它们经过精心挑选，只展示模型最强的一面。

你的测试应该回答三个问题：

```
边界问题1：这个能力在我的场景中，最差表现是什么？
  操作：用你最难的、最边缘的、最乱的真实数据测试
  判断：如果最差表现你能接受，那平均表现只会更好

边界问题2：这个能力的"断崖"在哪里？
  操作：逐步增加难度（更长的输入、更复杂的逻辑、更模糊的指令），找到断崖
  判断：你的日常使用场景在断崖之上还是之下？

边界问题3：这个能力的一致性如何？
  操作：同样的输入跑3次，看产出的方差
  判断：如果需要精确输出的场景，方差大=不可用
```

### 第3步：ROI估算——这个能力在我的工作流中能省多少时间？

不要凭感觉。用数字：

```
ROI = (旧方式耗时 - 新方式耗时) × 使用频次 - 学习成本 - 验证成本

例子：新Agent能力"自动代码审查"
- 旧方式：人工审查30分钟/次
- 新方式：AI审查5分钟 + 人确认10分钟 = 15分钟/次
- 使用频次：每周5次
- 每周节省：75分钟
- 学习成本：首次配置30分钟
- 验证成本：前两周每次多花5分钟确认 = 50分钟
- 首月净节省：75×4 - 30 - 50 = 220分钟
- ROI：显著正向 → 值得投入
```

**ROI为负不代表能力差。** 可能只是你的使用频次太低。AI辅助写遗嘱——能力很强，但你一年只用一次，ROI就低。AI辅助写周报——能力一般，但你每周用，ROI就高。

### 第4步：校准周期——多久重新评估一次？

AI能力在快速变化。三个月前"不行"的，今天可能"行了"。反之亦然——三个月前"惊艳"的，今天可能已经被更好的方案取代。

| 能力类型 | 建议重评估周期 | 原因 |
|:---|:---:|:---|
| 模型基础能力（推理、编码等） | 每次新模型发布 | 代际提升可能在一次发布中发生断崖式变化 |
| 工具/产品能力 | 每月 | 产品迭代快，月度更新可能包含关键功能 |
| Agent/编排能力 | 每两周 | 这是目前迭代最快的领域 |
| 垂直领域能力（法律、医疗等） | 每季度 | 垂直领域变化慢，受监管约束 |

## 第三层：个人AI能力雷达——一个活文档

把前两层结合起来，维护一个个人AI能力雷达。这不是"AI行业全景图"（太大，维护不了），而是"对我有价值的AI能力地图"。

### 雷达的四个象限

```
        高频使用
            |
    ┌───────┼───────┐
    │ 核心区 │ 探索区 │
    │ (深耕) │ (实验) │
    │       │       │
成熟 ─┼───────┼───────┤─ 新兴
    │       │       │
    │ 惰性区 │ 观望区 │
    │ (保持) │ (记录) │
    └───────┼───────┘
            |
        低频使用
```

**核心区（高频+成熟）**：你已经熟练掌握、每天都在用的能力。比如用Claude Code做代码编写和审查。
- 策略：深耕优化。写更好的记忆文件、更精细的Agent配置。
- 回顾频率：每月一次"有没有新的最佳实践？"

**探索区（高频+新兴）**：你每天都在做但还在探索AI怎么帮的事。比如投资分析中的自动研报交叉审查。
- 策略：主动实验。每周花1-2小时测试新能力在这个区的应用。
- 回顾频率：每周一次"这周测试了什么？"

**惰性区（低频+成熟）**：AI已经很擅长但你不需要经常做的事。比如AI翻译文档。
- 策略：保持现状。不需要优化，用的时候知道有就行。
- 回顾频率：每季度一次"是否有更好方案？"

**观望区（低频+新兴）**：AI的炫酷新能力但和你关系不大。比如AI视频生成——如果你不做视频内容，就不值得投入时间。
- 策略：记录+等待。看到新闻时记一笔，不投入时间。
- 回顾频率：每季度一次"是否进入了我的使用范围？"

### 雷达更新节奏

每周15分钟就够了：
1. 扫一遍四个信息源（5分钟）
2. 更新雷达——新能力进入哪个象限？（5分钟）
3. 决定本周要测试什么（5分钟）

## 第四层：从发现到内化——一个完整的测试循环

发现一个可能对你有价值的新AI能力后，不要"等有空了试试"。用这个循环：

### Day 1：15分钟快速验证（不通过就放弃）

```
用你的真实数据/场景跑一次。
只回答一个问题：这个能力在我的场景中，下限表现我能不能接受？
  → 能 → 进入Day 2
  → 不能 → 记录结果和原因，三个月后重看
```

### Day 2：30分钟深度边界测试（找到能力的真实范围）

```
用最极端的数据跑3次。
找到：
  1. 什么情况下表现断崖式下降？
  2. 一致性能接受吗？
  3. 和现有方案比，差异具体在哪里？
```

### Day 3：整合到工作流

```
如果前两步通过：
  1. 写一份简短的使用规范（什么是这个能力擅长/不擅长的）
  2. 在你的工作流中找一个真实任务，用新能力做
  3. 对比新旧产出，决定是"替代旧方式"还是"作为补充"
```

**三天、一小时。** 这比"先收藏以后看"有价值100倍。因为70%的"以后看"永远不会看。而这一小时，要么让你发现一个能省你大量时间的能力，要么让你在未来三个月不用再惦记它。

## 如果只能做一件事

**建一个个人AI能力雷达。** 用四个象限把自己的AI使用情况可视化。每周花15分钟更新。

这个动作看似简单，但它强迫你做一件事：**把"AI现在能做什么"这个模糊的焦虑，变成"对我有价值的AI能力有哪些，分别在什么阶段"这个清晰的地图。**

焦虑来自模糊。清晰消除焦虑。
