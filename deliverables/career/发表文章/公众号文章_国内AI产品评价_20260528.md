# 评价国内主流AI产品

> 陈颖芳 | 2026年5月
>
> 本文约4500字，阅读时间约12分钟。本文仅代表作者个人观点。
>
> 上一篇用10维评分体系评价了6款国际主流Agent产品。这一篇把镜头拉回国内——用同一套框架，评价国内用户真正能用的AI产品。

---

## 为什么国内产品需要单独评价

三个现实原因：

**第一，可用性鸿沟。** Claude Code、Devin、Cursor——这些产品的完整能力在国内需要科学上网、海外支付、英文界面。对大多数国内用户来说，"能不能用"比"好不好用"更优先。

**第二，生态差异。** 国内AI产品深度集成本地生态——微信、飞书、钉钉、WPS——这些集成场景在国际产品中完全不存在。飞书内嵌的AI助手在工作流中的价值，不是一个独立的Chat界面可以替代的。

**第三，使用模式不同。** 国内用户更习惯"All-in-One"的超级App模式，而不是CLI+脚本的开发者模式。这决定了产品的交互范式——国内产品偏向"聊天式AI"，国际产品偏向"工具式Agent"。

评价框架沿用上一篇的10维评分体系，但增加一个中国特有维度。

---

## 评价框架：10+1维

沿用前一篇的10维，每维1-5分，新增第11维：

| # | 维度 | 5分标准 |
|:---|:---|:---|
| 1-10 | 同前一篇（记忆/Agent编排/验证层/自主性/多模态/学习闭环/团队化/异常聚焦/环境隔离/开放性） | 同前一篇标准 |
| 11 | **本土生态集成** | 深度集成国内主流工作流工具（微信/飞书/钉钉/WPS/国产IDE） |

## 产品一：DeepSeek（深度求索）

> 2025年初以DeepSeek-V3和R1震撼全球，目前是国内技术实力最强的基座模型之一。产品形态：Web Chat + App + API。

### 逐维评价

| 维度 | 评分 | 理由 |
|:---|:---:|:---|
| 记忆系统 | ★★☆☆☆ (2) | 对话历史保存，但无结构化持久记忆、无自定义规则文件 |
| Agent编排 | ★★☆☆☆ (2) | API支持Function Calling，但Chat界面无多Agent能力 |
| 验证层 | ★☆☆☆☆ (1) | 无内建验证 |
| 自主性 | ★★☆☆☆ (2) | 仅问答模式，无自主执行 |
| 多模态执行 | ★★☆☆☆ (2) | 文件上传+图片理解，不支持浏览器/Shell操作 |
| 学习闭环 | ★☆☆☆☆ (1) | 无 |
| 团队化 | ★☆☆☆☆ (1) | 纯个人产品 |
| 异常聚焦 | ★☆☆☆☆ (1) | 无 |
| 环境隔离 | ★☆☆☆☆ (1) | 无 |
| 开放性 | ★★★★☆ (4) | 开源模型+低价API+兼容OpenAI格式，生态最好 |
| 本土生态集成 | ★★☆☆☆ (2) | 无明显生态集成 |

**综合：19/55（管理模式成熟度 35%）**

### 一句话诊断

**最好的国产基座模型，但不是Agent产品。** DeepSeek的核心价值在模型能力（推理强、编码强、中文强、便宜），不在产品形态。把它看作"中国版GPT-4级别的API"，用来给其他Agent产品提供算力，而不是把它当Agent用。

**最适合的场景**：作为Claude Code的替代模型后端、本地私有化部署的基座、中文长文本处理。

## 产品二：通义灵码（阿里云）

> 阿里云推出的AI编码助手，深度集成VS Code和JetBrains，背后是通义千问模型。

### 逐维评价

| 维度 | 评分 | 理由 |
|:---|:---|:---|
| 记忆系统 | ★★☆☆☆ (2) | .lingma/配置目录+项目级规则，但不如CLAUDE.md结构化 |
| Agent编排 | ★★☆☆☆ (2) | 单Agent，无并行编排 |
| 验证层 | ★★☆☆☆ (2) | 内建代码审查+单测生成 |
| 自主性 | ★★☆☆☆ (2) | Tab补全+问答，无自主多步执行 |
| 多模态执行 | ★★★☆☆ (3) | VS Code/JetBrains深度集成+终端，IDE内体验流畅 |
| 学习闭环 | ★★☆☆☆ (2) | 上下文学习但无跨会话持久化 |
| 团队化 | ★★★☆☆ (3) | 支持企业版+团队配置+代码规范统一管理 |
| 异常聚焦 | ★☆☆☆☆ (1) | 无 |
| 环境隔离 | ★★☆☆☆ (2) | 本地运行，无沙箱 |
| 开放性 | ★★☆☆☆ (2) | 仅限阿里云生态 |
| 本土生态集成 | ★★★★☆ (4) | 阿里云全家桶+钉钉+云效DevOps，国内最完整的开发工具链 |

**综合：25/55（管理模式成熟度 45%）**

### 一句话诊断

**国内IDE集成最好的编码助手，但远未到Agent能力。** 通义灵码的定位是"编码辅助"而非"任务执行"。它在代码补全、代码解释、单测生成上表现扎实，但它不会帮你规划任务、拆解步骤、编排多Agent。从管理模式视角看，它处于"工具辅助"（L1）阶段，离"任务委托"（L2）还有距离。

**最适合的场景**：阿里云生态内的日常编码、代码审查、单元测试生成。

## 产品三：Kimi（月之暗面/Moonshot AI）

> 以超长上下文（200万字）闻名，产品形态：Web + App + 浏览器插件。国内用户量最大的AI助手之一。

### 逐维评价

| 维度 | 评分 | 理由 |
|:---|:---|:---|
| 记忆系统 | ★★★☆☆ (3) | "Kimi记忆"功能——可记住用户偏好和关键信息，国内产品中记忆最好的 |
| Agent编排 | ★☆☆☆☆ (1) | 无Agent能力 |
| 验证层 | ★☆☆☆☆ (1) | 无 |
| 自主性 | ★★☆☆☆ (2) | 纯问答模式，但长上下文使它能"记住"整本书的内容来回答 |
| 多模态执行 | ★★☆☆☆ (2) | 文件上传+URL抓取+图片理解，但无Shell/浏览器操作 |
| 学习闭环 | ★★☆☆☆ (2) | "记忆"功能有学习能力但较基础 |
| 团队化 | ★☆☆☆☆ (1) | 个人产品 |
| 异常聚焦 | ★☆☆☆☆ (1) | 无 |
| 环境隔离 | ★☆☆☆☆ (1) | 无 |
| 开放性 | ★★☆☆☆ (2) | API可用但生态不如DeepSeek |
| 本土生态集成 | ★★★☆☆ (3) | 微信小程序+浏览器插件+App |

**综合：18/55（管理模式成熟度 33%）**

### 一句话诊断

**问答模式的极致——长上下文记忆是独特优势，但不是Agent。** Kimi的200万字上下文本质上是一种"会话内记忆"，这在处理超长文档时非常有用（一份500页的招股书可以完整丢进去）。但它的定位是一个"超级阅读理解器"，不是一个"任务执行器"。从管理模式视角看，Kimi是一个信息处理能力极强的工具，但如果要让它按你的规范、拆解任务、并行执行——它做不到。

**最适合的场景**：超长文档阅读、研报/财报全文分析、知识提取和总结。

## 产品四：豆包MarsCode（字节跳动）

> 字节推出的AI编程IDE，基于豆包大模型。产品形态：独立IDE（基于VS Code Fork）+ 浏览器版。

### 逐维评价

| 维度 | 评分 | 理由 |
|:---|:---|:---|
| 记忆系统 | ★★☆☆☆ (2) | 项目上下文感知，但无持久记忆系统 |
| Agent编排 | ★★☆☆☆ (2) | 多步代码修改能力，但不是多Agent编排 |
| 验证层 | ★★☆☆☆ (2) | 代码审查+错误修复 |
| 自主性 | ★★★☆☆ (3) | 可自主完成多文件代码修改和部署预览 |
| 多模态执行 | ★★★☆☆ (3) | 独立IDE+内置终端+预览+云部署 |
| 学习闭环 | ★★☆☆☆ (2) | 无持久学习 |
| 团队化 | ★★★☆☆ (3) | 云端项目共享+协作 |
| 异常聚焦 | ★☆☆☆☆ (1) | 无 |
| 环境隔离 | ★★★☆☆ (3) | 云端沙箱+独立IDE环境 |
| 开放性 | ★★☆☆☆ (2) | 封闭在字节生态内 |
| 本土生态集成 | ★★★★☆ (4) | 字节系全家桶（飞书+火山引擎+ Lark）+ 中文开发体验好 |

**综合：27/55（管理模式成熟度 49%）**

### 一句话诊断

**国内最接近"Agent化"的编码工具，独立IDE策略降低了配置门槛。** MarsCode的策略很聪明——不像通义灵码做IDE插件，而是做一个完整的AI IDE（类似Cursor的路，但是独立产品）。这给了它更大的自主性——不只是补全和问答，而是帮你创建项目、写多文件代码、部署预览。从管理模式视角看，它在从L1（工具辅助）向L2（任务委托）过渡。

**最适合的场景**：中文开发者快速原型开发、飞书生态内应用开发、不需要复杂Agent编排的个人项目。

### 一句话诊断

**最适合的场景**：需要实时搜索信息的问答、百度生态内的文档/网盘处理。

|:---|:---|:---|
| 记忆系统 | ★★☆☆☆ (2) | 基础记忆 |

| 多模态执行 | ★★★☆☆ (3) | 手机端操作+代码+图文 |
| 学习闭环 | ★☆☆☆☆ (1) | 无 |
| 团队化 | ★★★☆☆ (3) | 企业版+私有化部署+MaaS平台 |

**最适合的场景**：AutoGLM——手机端自动化操作（订票、填表、信息查询）；CodeGeeX——基础代码补全；企业私有化部署。

## 国内 vs 国际：差距在哪里

将国内6款产品的均分与国际6款产品对比：

| 维度 | 国际产品均分 | 国内产品均分 | 差距 |
|:---|:---:|:---:|:---:|
| Agent编排 | 2.8 | 1.8 | **-1.0** 差距最大 |
| 验证层 | 2.2 | 1.2 | **-1.0** 差距最大 |
| 自主性 | 3.2 | 2.3 | -0.8 |
| 学习闭环 | 2.2 | 1.5 | -0.7 |
| 多模态执行 | 3.0 | 2.5 | -0.5 |
| 环境隔离 | 2.8 | 1.7 | -1.2 |
| 记忆系统 | 2.7 | 2.2 | -0.5 |
| 开放性 | 3.2 | 3.0 | -0.2 差距最小 |
| 团队化 | 2.5 | 2.2 | -0.3 |
| 本土生态集成 | — | 3.3 | 国内独有优势 |

### 差距最大的三项

**Agent编排（1.8 vs 2.8）**：国内产品几乎全部停留在单Agent或无Agent阶段。没有任何一款产品支持"同时启动多个Agent并行执行不同维度的任务"。这是管理模式和生产效率被拉开的核心原因。

**环境隔离（1.7 vs 2.8）**：国内产品几乎没有沙箱/隔离概念。MarsCode的云端沙箱是唯一的例外。

**验证层（1.2 vs 2.2）**：国际产品至少开始有交叉审查和自动验证的基础能力，国内产品在这个维度近乎空白。

### 国内的优势

**本土生态集成（3.3，国际产品此项无意义）**：飞书/钉钉/微信/WPS的深度集成是国际产品无法复制的。一个飞书消息就能触发的AI助手，在工作流中的实际使用频率远超需要打开终端的CLI工具。

**开放性（3.0 vs 3.2，差距最小）**：DeepSeek和智谱的开源+低价API策略使国内在模型可获取性上不输国际。

## 一个现实建议

如果你在国内，想要"管理模式"级别的AI协作体验，当前最务实的路径是：

**用国产模型作为基座 + 用国际产品的Agent框架。**

具体来说：
- **基座模型**用DeepSeek API（便宜、中文强、推理好）
- **Agent工具**用Claude Code（通过API代理）+ 自己写的Prompt模板
- **长文档处理**用Kimi（200万字上下文独步天下）

这不是一个产品，而是一个组合。但在国内产品的Agent能力赶上之前，这是唯一能同时享受"中文模型能力"和"Agent编排能力"的方式。

**等到国产产品在Agent编排、环境隔离、验证层这三个维度上各提升1-2分——也就是2-3个版本迭代——这个组合策略就需要重新评估了。** 到时候，MarsCode或AutoGLM可能已经填上了Agent能力的缺口。你在国内的AI工作流，可能只需要一个产品。
