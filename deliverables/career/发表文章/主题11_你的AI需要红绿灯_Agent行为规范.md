# 你的AI需要红绿灯——Agent行为规范

> 2026年6月3日，微软发布了Agent Control Specification (ACS)——全球首个开源AI Agent行为约束标准。这不是一个技术新闻，它告诉我们一件事：AI Agent跑得太快了，是时候给它们装上红绿灯了。

---

## 引子：AI Agent有了"交通法规"

你有没有遇到过这种情况——

你让AI Agent帮你整理投资文档，它一口气读完了所有文件，写了分析报告，然后…顺手删了源文件的一行注释。你没发现。三天后你发现数据对不上，追了三天的根因，最后发现是那行被删的注释里藏着一个时间偏移配置。

我的第一反应不是生气，是后怕。如果它删的不是一行注释，而是一个关键表格呢？

这不是假设。2026年6月，AI Agent的能力已经从"问答助手"进化到了"多步骤自主执行者"——读文件、写代码、跑脚本、调API、发消息。能力越大，越界的机会也越大。

就在这个时间点，微软Build 2026大会上发布了**Agent Control Specification (ACS)**——这套标准定义了Agent该做什么、不该做什么、做了什么怎么记、做错了怎么停。本质上，它是AI Agent的第一部"交通法规"。

---

## 一、AI Agent的"交通事故"——为什么需要行为规范

### 真实场景

在过去一个月使用Claude Code的过程中，我经历过这些"差点出事"时刻：

- Agent在整理文档时，同时修改了我没想让它们改的文件。
- 一次大规模文件操作中，Agent问我"是否要删除这个文件"——那个文件恰好是某个关键引用的源。
- Agent在执行Shell命令时，曾请求运行一个我完全没预料到的命令。
- 在一次投资分析中，Agent读取的信息超出了我打算给它看的范围。

每一种都不是恶意，都是"我觉得这样更好"或"我猜你需要这样"。这正是问题所在——**AI Agent没有"坏"的意图，但它也没有"好"的边界感。**

### 为什么工具越强，风险越大

早期AI（ChatGPT 3.5/4.0时代的"问答模式"）没有这个问题——它只能回答，不能执行。它的所有"行为"都停留在你的聊天窗口里。

但现在你用的AI Agent有工具：读/写文件、跑Shell命令、调API。每个工具都是有用的能力，也是潜在的"越界工具"。不是AI变危险了，而是**AI能做的事变多了，但"不能做的事"还没定义清楚。**

这就像给一个新手司机一辆法拉利，只教了油门，没教刹车和红绿灯。

## 二、从ACS到行为规范：红绿灯、摄像头和交警

微软ACS实际定义了一套基于"5个检查点"的技术框架——在Agent的输入、大模型调用、状态改变、工具执行、输出五个环节设置安全策略（YAML策略文件声明式管控）。但对我个人使用者来说，一套更直觉的抽象是：AI Agent的行为规范只做三件事——设定它不能做什么、记录它做了什么、在关键节点让人来决定。

这三个维度恰好对应交通系统的三个元素。

### 🚦 红灯：权限分级——Agent能做什么、不能做什么

交通系统中，红灯停、绿灯行、黄灯缓行。受ACS的检查点机制启发，我按操作的风险等级给Agent上不同的"灯"：

| 级别 | 名称 | 定义 | 默认策略 |
|:----:|:----|:----|:--------|
| **L0** | Read | 读文件、搜索内容 | 自动执行 |
| **L1** | Write | 创建/修改文件 | 自动+通知 |
| **L2** | Execute | 跑命令、调API | 每次确认 |
| **L3** | Delete | 删除/覆盖/不可逆 | 双重确认 |

**为什么这有用？**

因为不同操作的不可逆程度完全不同。读文件可以反悔吗？不能，但也没破坏性。删除文件可以反悔吗？理论上可以（Git恢复），但如果你没有在Git里追踪——那就永远找不回来了。

权限分级的作用不是"限制AI"，而是**让你在给AI指令时想清楚一件事：这个任务，AI可以做到什么程度？**

### 📹 绿灯：审计追溯——Agent做了什么、怎么做的

交通系统中的摄像头不是用来"阻止违章"，而是用来"记录违章"——知道自己被记录的人，行为更规范。

在AI协作中，每个Agent操作都应留下可追溯的记录链：

```
Agent操作 → 时间 → 类型 → 影响文件 → Git Hash → 审查结果
```

这个链条解决了一个实际问题：**当AI做错了事，你能多快找到根因。**

以我的实际体验为例——有一次我发现某篇公众号文章的封面图链接断了。如果没有Git记录，我需要回忆"那天改了哪个文件"——AI会话记录可能有几百条token交互。但因为有Git commit记录，我三分钟就找到了那个commit："2026-05-30 17:33 | 修改封面图引用路径"。

这种"追踪能力"的价值不只在出了问题的时候。当你发现追踪系统存在时，你和AI的协作关系就从一个"模糊的信任"变成"有据可查的协作"。

### ✋ 黄灯：人工干预——什么需要人确认、什么自动停

第三个框架是最重要的：**人不是AI操作的旁观者，而是关键节点的决策者。**

| 场景 | 需要什么干预 |
|:----|:-----------|
| 删除文件 | 双重确认 |
| 投资决策 | 双重确认+独立评分 |
| 对外发布文章 | 审阅+确认 |
| 修改≥3个文件 | 先看计划再执行 |
| 重复失败/资源消耗激增 | 自动暂停 |

注意一个关键点：**不是所有操作都需要人干预。**

如果你让AI帮你搜索最近的AI新闻（L0 Read），每次都要你确认"我可以搜索吗"，你会疯掉。但如果AI说"我要删除你投资目录下的三个文件"（L3 Delete），而你没看一眼就点了"允许"——那问题不在AI，在人。

好的AI Agent行为规范，应该在"太松"（什么都能做，出了事你都不知道）和"太紧"（每步都问你，根本无法工作）之间找到平衡点。

## 三、个人版Agent行为规范——今天就能做的三件事

上面说的都是"框架"。以下是你**今天**就能在Claude Code里做的事。

### 第一件事：给每个任务标注权限级别

在每次启动新任务时，在指令开头加入一句权限声明：

```
❌ 不写权限：帮我整理投资文档（AI不知道该读到什么级别）
✅ 写权限：帮我整理投资文档。L1权限——可以读写，但不要执行脚本
```

这不是多余的流程。你告诉AI"你可以做什么"和"你可以做到什么程度"是两回事。前者解决了"做什么"，后者定义了"不能过什么线"。

### 第二件事：用Git作为你的审计追溯系统

你可能已经在用Git管理代码。但你在用Git管理"AI做了什么"吗？

方法很简单：每次AI产出和修改后，看一眼变更摘要再commit。

这一步不需要额外工具，就是Claude Code自动Git commit前的"看一眼"环节。但这一眼的价值巨大——它让你在commit之前有一次"慢下来"的机会，逐一检查AI改了什么。

### 第三件事：设置你的"双重确认"规则

列出你"绝不自动"的操作清单。我的清单包括：

- 删除任何文件（尤其是非临时文件）
- 任何涉及资金/投资的操作
- 公众号文章的发布（可以写，但发布前必须人看一遍）
- 修改CLAUDE.md或关键规则文件

对于这些操作，Claude Code的权限弹窗本身就是一种"确认门"——关键是不要自动放行，而是要**真正停下来看清楚**AI要做什么。

## 四、从个人到组织——为什么银行需要Agent行为标准

写到这里，你可能会想："这和我有什么关系？我又不写代码。"

但如果你在银行工作（比如和我一样在科技金融部门），Agent行为规范对你有两层意义。

### 第一层：AI Agent很快就会出现在银行的风险流程中

不是"如果"，而是"什么时候"。当银行的信用评估AI Agent、反欺诈AI Agent、客户服务AI Agent开始自主执行时——审批贷款、标记可疑交易、回复客户投诉——每一个动作都需要权限分级、审计追溯和人工干预机制。

**没有红绿灯的AI Agent，是一个不需要钥匙的自动行驶车辆。**

### 第二层：Agent行为规范会成为企业AI治理的基础标准

ACS开源标准的出现，意味着AI Agent行为管理正在从"每家自己摸索"走向"行业标准"。当这个标准被银行监管机构注意到时（大概率在未来12-18个月），遵循ACS逻辑的银行将在合规评估中领先一个身位。

**你今天在自己工作中建立的Agent行为习惯——权限声明、变更审查、操作记录——就是银行级AI治理的微型原型。**

### 五个立即行动

无论你是个人用户还是银行从业者：

1. **今天** — 检查你的AI工具当前有哪些"它想做什么就做什么"的默认配置
2. **本周** — 列出你的"三级权限清单"（可以自动/需要确认/必须双重确认）
3. **本月** — 确保所有重要AI产出都有Git记录（或用你习惯的工具做操作日志）
4. **本季度** — 回顾一次"AI差点越界"的时刻，更新你的权限规则
5. **本年度** — 如果贵行正在讨论AI治理，把ACS标准作为一个参考框架

---

## 结尾：AI越强，红绿灯越重要

2026年6月的AI世界，比2025年强大得多——更强的模型、更长的上下文、更自主的Agent、更多的工具。但能力增长带来一个结构性问题：**AI能做的事越来越多，但"不能做的事"边界定义跟不上。**

这不是AI的问题，这是我们——使用者——的问题。我们太关注"AI能做什么"，太少关注"AI不能做什么"。

微软ACS标准的价值，不在于它是一个技术规范，而在于它传递了一个信号：**AI Agent需要红绿灯，而这条规则掌握在人的手里。**

权限分级是红灯——停了该停的。
审计追溯是绿灯——记录该记的。
人工干预是黄灯——在不确定的时候，停下来问问人。

你的AI越强，红绿灯就应该越清晰。不是限制AI，是让协作更可控、更可持续。
