# AI-Native Startup 方法论

> 基于 Anthropic《The Founder's Playbook: Building an AI-Native Startup》（36页原文）| 2026-06-02 v2
> 适用场景：一人公司(OPC) / 小团队AI Native创业

---

## 零、核心理念

**创始人角色的根本转变**：

```
传统创始人 = 亲自干活的人        → 瓶颈是自己
AI时代创始人 = 设计系统的人       → 瓶颈是判断力
```

**关键约束变迁**：
```
从：你能建什么 → 到：你选择建什么
执行力便宜了，判断力变得稀缺
```

**三个Claude工具的分工**（原文反复强调，不可混淆）：

| 工具 | 什么时候用 | 为什么 |
|:-----|:-----------|:-------|
| **Chat** | 快速问答、头脑风暴、魔鬼代言人 | 对话式，零设置 |
| **Claude Code** | 写代码、建产品、操作代码库 | 代码库访问+git+Plan Mode |
| **Claude Cowork** | 自动化运营、CRM、报告编排 | 文件夹访问+连接器+定时运行 |

> 你的14个自动任务+5个Agent = 你用Claude Code+Cron替代了Cowork

## 一、四阶段框架（含出口标准+核心危险+关键练习）

### 阶段1：Idea（验证问题）

**目标**：Problem-Solution Fit — 先验证问题，再写代码

**出口标准**（三个都答"是"才推进）：
1. 问题是否真实且具体？——你能说清楚谁遇到、多频繁、多严重
2. 解法是否解决了验证过的问题？——不是你以为的问题，是验证揭示的问题
3. 有足够信号支持建MVP吗？——永远不会有确定性，但有定性证据就行

**核心危险**（原文强调三个）：
```
① 把"建出来"当成"验证过"（42% startup失败是因为建了没人要的东西）
   AI让原型从想法到产品的时间压缩到极致 → 这个比例只会更高

② 确认偏误放大（问AI"我的想法对吗" → AI会找到支持证据）
   解法：用AI做"结构化魔鬼代言人"——强制它论证你错在哪里

③ 过早规模化（建得太快太容易，还没验证就铺开了）
   解法：感知先行于建设——保持"建前先想清楚"的纪律
```

**AI实践**：

| 任务 | 用哪个Claude | 做什么 |
|:-----|:------------|:-------|
| 问题假设验证 | Chat | 魔鬼代言人，找反证 |
| 市场研究 | Chat+Cowork | TAM/SAM/SOM模型，竞品地图 |
| 客户发现 | Chat | 设计访谈问题，分析访谈记录 |
| 轻量原型 | Code | 只建核心交互，给5个人看 |

**原文关键练习**（我简化了，但保留了核心）：
> 每次访谈后问Chat：哪些证据支持假设，哪些挑战假设。如果前者远多于后者，问自己——这是数据真实分布，还是我选择性地看到了想看的？

### 阶段2：MVP（建什么）

**目标**：Product-Market Fit 早期信号

**出口标准**：有真实证据表明一群特定用户觉得产品足够有价值→愿意回来用、付费、或告诉别人。

**核心危险**：
```
① AI技术债（最独特的危险）
   没有架构约束的AI每轮对话重新推导基础决策
   → 代码库没有一致的思维模型
   → 每个新session都在重建，而不是叠加
   解法：写CLAUDE.md，每次session开始回顾+结束更新

② 零摩擦范围蔓延
   每个功能单独看都有道理，但加在一起产品就失去了边界
   因为传统制约（开发成本）不存在了
   解法：建scope文档+功能准入标准

③ 误把早期势头当PMF
   ProductHunt流量/朋友的支持/投资人的热情 ≠ PMF
   Sean Ellis测试：>40%用户"没了你会很失望"=真信号
```

**AI实践**：

| 任务 | 用哪个Claude | 做什么 |
|:-----|:------------|:-------|
| 架构定义 | Chat | 建CLAUDE.md：原则/依赖/取舍 |
| 范围控制 | Chat | 写scope文档+准入标准 |
| 核心构建 | Code | 写代码、测试、调试 |
| 安全审查 | Code | 首次安全扫描再上线 |
| 用户反馈 | Cowork | 安排访谈、追踪bug、周报合成 |

**原文最实用的建议**：
> 建一个session模板：每次打开Claude Code前，（1）回顾scope文档（2）提供CLAUDE.md架构上下文（3）会话结束时花5分钟更新日志。5分钟/次是防止架构漂移的廉价保险。

### 阶段3：Launch（增长）

**目标**：可规模化的复利增长，可衡量的CAC/LTV/回收期

**出口标准**（三个全满足）：
1. 增长是可重复的、渠道驱动的 —— 知道用户从哪来、花多少钱、值多少钱
2. 产品能扛生产负载 —— 安全+合规+可靠性已就绪
3. 运营不依赖创始人 —— 支持、排期、报告已系统化，不需要你亲自处理

**核心危险**：
```
① 创始人成为瓶颈（最难跨越的坎）
   MVP阶段你在每个环节是资产 → Launch阶段变成约束
   "从自己做"到"设计系统让别人做"是创业生命周期最难的转变之一
   信号：一个决定从1小时变成1周才能轮到你

② 技术债到期
   MVP阶段积累的债从"利息"变成"本金"
   需要系统性的架构审计+针对性重构+测试覆盖

③ 安全合规不再可推迟
   有真实用户+真实数据+企业合同 → 安全漏洞从理论风险变成真实暴露
```

**你的Launch操作系统**（原文说的你已经在跑）：
```
6:37  P0推进Agent      → 扫描六目标 → 推送微信
7:00  AI搜索Agent      → 简报搜索
8:00  文章发布检查      → 自动化
8:37  综合简报生成      → 自动化
9:03  早间推送          → 自动化
14:00 AI搜索（午）     → 自动化
21:00 AI搜索（晚）     → 自动化
21:07 晚间推送          → 自动化
22:42 自动commit        → 自动化
每周六 7飞轮扫描        → 系统化检视
```

这就是原文说的"Launch阶段操作系统"——你已经跑起来了。

### 阶段4：Scale（可持续）

**目标**：可持续运营——盈利/IPO/被收购

**出口标准**：公司可持续运营，即使创始人不再直接参与日常运作

**核心危险**：
```
① 放手的心理障碍
   从"建系统"到"信任系统"——比建系统更难

② 组织能力跟不上
   GTM（Go-To-Market）功能需要从"创始人卖"升级为"系统卖"

③ 护城河不够深
   一个资金充足的对手复制你的产品——用户会留下吗？
```

**构建护城河的三种路径**（原文详细论述）：

| 护城河类型 | 原文表述 | 你的资产 |
|:-----------|:---------|:---------|
| **领域知识** | 隐性行业经验，通用模型无法复制 | 20年银行+科技金融方法论 |
| **数据飞轮** | 用户行为信号，竞争对手买不到 | Agent使用中的决策数据 |
| **工作流锁定** | 用户的切换成本=重建工作方式 | 14个自动化任务+5个Agent |

**原文最有力的洞察**：
> 10人团队 = 过去100人的产出。Medvi（4.01亿美元营收，仅2名全职员工）不是例外，是预告。

### 3.2 原文对你最有用的三条建议

1. **建CLAUDE.md** — 你已经在用。每次session开始回顾+结束更新5分钟日志。

## 四、关键提醒（来自原文结尾）

> "在AI时代，创始人的工作没有变——找到一个真实问题，建一个解决它的东西，把它规模化为一个有意义的公司。
>
> 变的是到达那里的路径：
> * 验证周期从几个月压缩到几个下午
> * 原型不再需要联合创始人，只需要一个清晰的问题+几次聚焦的会话
> * Launch从发布前的冲刺变成持续的工作流
> * Scale阶段过去需要新人救火的运营负担可以交给AI
>
> **瓶颈不再是你能建什么，而是你选择建什么。** "
