# AI-Native Startup 方法论

> 基于 Anthropic《The Founder's Playbook: Building an AI-Native Startup》| 2026-06-02
> 适用场景：一人公司(OPC) / 小团队AI Native创业

---

## 零、核心理念

**创始人角色的根本转变**：

```
传统创始人 = 亲自干活的人
AI时代创始人 = 设计系统的人
```

不是"公司+AI工具"，而是**从第一天起以AI为核心基础设施**。

**关键约束变迁**：

```
从：你能建什么 → 到：你选择建什么
执行力便宜了，判断力变得稀缺
```

---

## 一、四阶段框架

### 阶段1：Idea（验证问题）

**目标**：Problem-Solution Fit（先验证问题，再写代码）

**核心危险**：确认偏误——AI比任何时候都更擅长为坏想法编织漂亮的论证。一个跑起来的原型 ≠ 需求验证。

**你的实践**：

| AI工具 | 作用 |
|:-------|:------|
| Claude Chat | 结构化"魔鬼代言人"——强制它论证你的想法为什么是错的 |
| Web搜索+3票验证 | 多路平行搜索+对抗验证，不让自己被单一信息源说服 |
| Agent A-E | 用投资分析管道评估"这个方向值得投入吗" |

**出口标准**（三条都答"是"才推进）：
1. 问题是否真实且具体？
2. 你的解法是否真的解决了验证过的问题？
3. 你有足够的信号（不是确定性）吗？

### 阶段2：MVP（建什么）

**目标**：Product-Market Fit 早期信号

**核心危险**：技术债 + 范围蔓延。AI让加功能变得零摩擦，导致代码库缺乏一致的思维模型。

**关键实践**：
1. **先写 CLAUDE.md** — 在写任何代码之前，记录架构原则、设计取舍、范围边界
2. **模块优先** — 用Claude Code先生成模块框架，再填充功能
3. **Sean Ellis测试** — >40%用户"没有你的产品会非常失望" = PMF信号

**你的CLAUDE.md应该包含**：
```markdown
# 项目规则

## 核心原则
- 只用[功能列表]的最小可行
- 不添加"以后可能会用到"的东西

## 架构决策
- 技术栈：[你的选择]
- 数据流向：[描述]
- 不做什么：[边界清单]
```

### 阶段3：Launch（增长）

**目标**：可规模化的复利增长，可衡量的CAC/LTV/回收期

**核心危险**：**创始人成为瓶颈**——最难跨越的坎是从"自己做"到"设计系统让别人/Agent做"。

**你的实践**：
- 建立Launch-stage操作系统：用Agent自动化日程、CRM、报告、内容生成
- 创始人聚焦于：产品决策、关键客户关系、融资
- 重复性任务 → Agent

**你已有的Launch系统**：
```
6:37  P0推进Agent → 扫描六目标 → 推送微信
7:00  AI搜索Agent → 简报搜索
8:00  文章发布检查
8:37  综合简报生成
9:03  早间推送
14:00 AI搜索（午）
21:00 AI搜索（晚）
21:07 晚间推送
22:42 自动commit
每周六 7飞轮扫描
```

这就是你的Launch操作系统——你已经跑起来了。

### 阶段4：Scale（可持续）

**目标**：可持续运营——盈利或IPO

**核心危险**：放手——创始人必须克服"不放心交给别人/Agent"的心理障碍。

**关键洞察**：
- AI消除了传统团队规模假设
- **10人团队可以产出过去需要100人的产出**
- 案例：Medvi（4.01亿美元营收，仅2名全职员工）

**构建护城河**：

| 护城河类型 | 你的资产 | 为什么难复制 |
|:-----------|:---------|:-------------|
| 领域知识 | 20年银行经验+科技金融方法论 | 通用模型无法复制的隐性知识 |
| 用户数据飞轮 | Agent使用中的行为数据 | 竞争对手买不到 |
| 工作流锁定 | 14个自动化任务+5个Agent | 切换成本是"重建工作方式" |
| AI方法论体系 | V1.0完整+286个文档 | 不是工具，是思维框架 |

## 二、OPC的AI Native实践

### 2.1 你的核心优势

```
传统OPC的约束：一个人无法覆盖开发/运营/销售/支持
AI Native OPC：一个人+AI = 一个团队
```

你已经验证了的AI Native能力：

| 传统团队分工 | 你的做法 | AI工具 |
|:------------|:---------|:--------|
| 开发 | 写模板+配置Agent | Claude Code |
| 运营 | 14个自动任务 | cron+Agent |
| 内容 | 公众号文章+知识卡片 | Claude辅助 |
| 研究 | Agent A-E分析管道 | 多Agent协作 |
| 知识管理 | 91张卡片+Obsidian | 系统化整理 |
| 投后跟踪 | 飞轮周扫描 | 定时扫描 |

### 2.2 你自己就是最佳案例

> 2026年5月，一个人+AI，30天产出：
> - 286个交付文档
> - 296个PDF
> - 14个自动任务
> - 5个协作Agent
> - 10篇公众号文章
> - 91张知识卡片
> - 科技金融方法论V1.0
> - 个人知识体系完整框架

这就是AI Native OPC的活案例。

## 四、关键提醒

来自Playbook的最重要三条：

1. **验证优先**。AI让建东西变得很快也很危险。不要把跑起来的原型当成需求证明。
2. **先写CLAUDE.md**。在你写任何代码之前，记录架构原则和范围边界。
3. **审计你的时间**。任何重复的、不需要你判断的事情 → 交给Agent。
