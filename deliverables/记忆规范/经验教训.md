## 一、规范建设

### 教训1：规范先行，实战滞后

**场景**（2026年5月，从0到1执行）：交叉审查SOP写好了、评分矩阵建好了、L1自查清单写好了——但都没跑过实战。Agent角色卡5个，没有一个在真实任务中使用过。

**根因**：写规范比跑实战容易得多。写完规范有"完成感"，但这种完成感会误导——资产未经实战验证，最多是L1.0。

**检测信号**：新增规范文件的速度 > 规范被实战使用的速度

**避免方法**：
- 每写完1份规范 → 在24小时内至少让它被1个真实任务检验
- 在每周自检中增加："本周写的规范，有几份被实战用了？"
- 新规范创建时，同步创建一个"验证任务"——明确什么任务会用到它

### 教训2：两套同类体系不关联→编号冲突

**场景**（2026年5月，审查发现）：五层质量网（L1-L5）和AI产出验证规则（L0-L4）是两套独立体系，编号使用相同的L1-L5前缀但含义完全不同。手册整合了质量网，但从未提及验证规则的存在。读者拿到两份文件会困惑"到底用哪套"。

**根因**：两个体系的创建时间相近（5/29、5/30），各自独立设计，整合时未交叉检查编号体系。

**检测信号**：两个文件各自定义了层级编号，编号前缀相同但含义不同

**避免方法**：
- 新建规范前，先检查已有规范中是否已有同名编号体系
- 如可能冲突，使用不同前缀（如Q1-Q5 vs L0-L4）
- 新增规范时在"上下文加载清单"中确认与新规范的关联关系

### 教训3：文件更新后正文遗漏→自相矛盾

**场景**（2026年5月，审查发现）：手册v1.2声称"v1.2更新：新增失败案例"，失败4明确纠正了"3-6个月"的时间估计，但正文2.3节的原句没有被修改。结果v1.2正文说"3-6个月"，失败4说"这个估计是错的"。

**根因**：版本更新时只加了新内容（失败案例），没有检查新内容和旧内容之间的逻辑一致性。

**检测信号**：一个文件内，两个段落对同一事实给出不同数字/结论

**避免方法**：
- 版本升级时，搜索旧内容的"关键词"（如"3-6个月"），确认它在全文中被一致改写
- 新增"纠正性内容"（如失败模式）时，在旧位置增加交叉引用——"见第十二章失败4"
- 规则：任何版本升级时，全文搜索本次要纠正的断言，确保无一遗漏

## 二、任务执行

### 教训4：规划解耦未形成肌肉记忆→"直接做"

**场景**（2026年5月）：规范要求"重要任务先出计划→审阅→执行"，但多数任务仍是直接执行模式。任务做到一半发现方向偏了。

**根因**：规范是新的（5/29），旧习惯是"发指令→等结果"。人类改变工作习惯需要时间，光靠"知道该做什么"不够。

**检测信号**：任务做到一半需要"回头改方向"或"推倒重做"

**避免方法**：
- 在CLAUDE.md中**硬编码**规则："影响≥3个文件的任务，必须先输出计划并等待确认"——不让自觉决定，让系统强制执行
- 把规划解耦的执行监督交给AI自己——"检测任务影响超过3个文件→主动要求先出计划"——不要依赖人记得

### 教训5：执行纪律——可验证承诺的完成率低

**场景**（2026年5月，首次飞轮扫描）：上周4项"下周行动"承诺，只完成了2项（50%）。未完成的2项（验证cron、多Agent并行）都是需要"主动去做"的事，而完成的2项（交叉审查、周反思）都在工作流中自然发生。

**根因**：承诺的完成依赖于"人记得去做"，而非"系统会在正确的时间提醒"。

**检测信号**：连续2周自检清单中同一项标记为"否"

**避免方法**：
- 每项"下周行动"标注触发条件——"当X发生时做Y"，而不是"下周某天做Y"
- 未完成的项自动滚动到下周，最多滚动2次。第3次仍然"否"→升级为深度诊断（启动模板C的🚨机制）
- 把可验证承诺的数量控制在≤5条

## 三、自动化与基础设施

### 教训6：定时任务配置了但未验证→"它应该在工作吧"

**场景**（2026年5月）：4个cron任务配置了双层保障（系统crontab + Claude Code CronCreate），但从未检查它们是否真的触发过。首次飞轮扫描时发现——操作日志目录中没有任何cron产出。

**根因**：配置完成有"完成感"，但这种感觉替代了"验证它真的在工作"。

**检测信号**：新基础设施配置完成后，24小时内无产出日志

**避免方法**：
- 任何定时任务配置后，**立即手动触发一次验证**
- 每日简报等任务应在创建后第一个触发时间点后**1小时内检查产出**
- 在每周飞轮扫描中增加："本周cron任务：计划执行次数 vs 实际执行次数"

### 教训7：脚本专项化过度→每个产出一个脚本

**场景**（2026年5月，手册失败3）：37个Python脚本中，PDF构建器占了13个——每个文档一个构建脚本。同一逻辑在多个脚本中重复定义。

**根因**：快速出成果的冲动——"先让这个能跑"→复制粘贴上次的脚本→积少成多。没有定期"合并重复→抽取共用"的纪律。

**检测信号**：同一个函数/逻辑在≥3个文件中出现

**避免方法**：
- 在每周自检中增加："本周新增脚本中有逻辑与已有脚本重复的吗？"
- 当同一函数在≥3个文件中出现→必须抽取到共用模块
- 每新增3个"专项脚本"→停下来做一次重构

## 四、AI协作行为

### 教训8：上下文过载——全量加载 ≠ 完整加载

**场景**（2026年5月，原则2补充）：手册强调"给AI完整上下文"，但早期实践中倾向于"把所有规范文件都装上"。结果：AI注意力分散，产出质量反而下降。

**根因**：把"完整"理解为"全量"。实际上，"完整"= "与本次任务相关的都给了"，不是"所有的都给了"。

**检测信号**：AI产出中出现与本次任务不相关的内容，或遗漏了关键约束

**避免方法**：
- 参照"上下文加载清单"——每次1-3份规范 + 触发条件的记忆
- 不确定该加载什么→宁可少加载1份核心规范，也比全部加载好
- 如果AI产出明显偏离→先检查"是不是上下文给太多了"

### 教训9：多Agent并行——编排力容易闲置

**场景**（2026年5月，首次飞轮扫描）：Agent角色卡库有5个角色，但连续多周无多Agent并行。任务能用单Agent完成，就不会主动拆成并行。

**根因**：多Agent并行需要"主动拆解任务"——这个动作不会自然发生。人会倾向于"一个Agent做完算了"，而不是停下来想"能不能并行"。

**检测信号**：连续2周飞轮自检中"编排力"项标记为"否"

**避免方法**：
- 每周至少1个任务明确拆成≥2个Agent并行
- 在收到复杂任务时，先问自己："这个任务可以拆成哪几个独立子任务？"
- 模板C已内置检查——"如果否：下周挑1个复杂任务，拆成≥2个Agent并行执行"

### 教训10：感知力——知道≠会用

**场景**（手册失败5）：每日AI简报持续更新，信息量充足，但新能力的追踪偏向被动接收——"看到一条AI新闻→记下来"，而非"这个新能力对我有什么用→我去试一下"。

**根因**：信息收集的"完成感"替代了真正的学习和实验。"我已经知道了"≠"我已经会用了"。

**检测信号**：连续2周感知力自评有"知道的新东西"但无"实际试过的新工具/方法"

**避免方法**：
- 每了解2个AI新能力 → 至少挑1个实际试试
- 在每周自检中追问："最重要的AI新发现，我试了吗？效果如何？"
